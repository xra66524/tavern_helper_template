## 一、界面定位与总体架构

### 1.1 叙事定位

USER 是「摄像机/观察者」，状态栏应呈现**群像全景**而非单一角色面板。94 名穿越者 + 305 节点科技树 + 扁平化机构表 + 多势力外部博弈 + 情报网络 + 事件队列，共同构成一个**多 Tab 群像仪表盘**。

### 1.1.1 整体布局方案：手机案卷 + 底部菜单（最终采用）

本状态栏**完全以手机为第一优先**，不再考虑电脑端左侧目录适配。原「横向 Tab + 单内容区」方案改为**手机 APP 式底部菜单 + 案卷正文区**：

```text
┌──────────────────────────────┐
│  QUNYOU CROSSING      LIVE   │
│  📅 1905-05-10  第1天 混乱期 │  ← 顶部世界案卷抬头
│  当前大事件：确认穿越处境     │
├──────────────────────────────┤
│  🚛 基地车  当涂县南岸        │  ← 基地车简报条
│  ⚛️正常  ⚡72%  🦾若干       │
├──────────────────────────────┤
│                              │
│  🔬 案卷 1/7 · 科技           │  ← 当前案卷正文
│  ┌────────────────────────┐  │
│  │ 本页摘要                │  │
│  │ 科技树面板施工中……      │  │
│  └────────────────────────┘  │
│                              │
│  ┌────────────────────────┐  │
│  │ 科技案卷                │  │
│  │ 后续接入 DAG / 进度……   │  │
│  └────────────────────────┘  │
│                              │
├──────────────────────────────┤
│ 🔬科技 🏛组织 ⚔军事 👥角色 🌐外部 │  ← 底部 APP 菜单栏
│ 🕵情报 ⚠事件 💰财政 🗺区划 📋人事 │     横向滚动，可选菜单项按数据存在显示
└──────────────────────────────┘
```

设计原则：

1. **底部菜单即目录**：方案 D 的「左侧目录」在手机上改造为底部固定/粘滞菜单，类似手机 APP 的 Tab Bar。
2. **正文是案卷页**：每个模块不是传统网页 Tab，而是一页「案卷正文」；顶部有案卷编号与摘要卡，后续组件作为案卷内容接入。
3. **不再支持折叠为空**：底部菜单点击即切换案卷，当前案卷始终有正文，避免手机端出现空白折叠态。
4. **手机第一**：宽度、小字号、底部菜单、纵向正文优先；电脑端只保证可用，不单独优化。
5. **叙事风格**：整体视觉从「现代控制台」偏向「1905 年情报案卷 + 基地车电子化抬头」混合风格。
6. **所有面板和弹窗必须内置编辑模式**（§三·五）：每个数据展示面板/弹窗都必须提供「✎ 编辑」切换入口，使 GM/玩家无需切回提示词即可修正 stat_data。这是硬性要求，非可选优化。只读展示且无 stat_data 写入需求的纯导航组件（如 `TabNav`）可豁免。

### 1.3 数据流

```text
stat_data (MVU 变量)
    │
    ▼
store.ts (defineMvuDataStore, 2s 轮询 + watchIgnorable 双向同步)
    │
    ▼
App.vue (useDataStore() → 注入各子组件)
    │
    ├── 静态结构 join：TechTreePanel 读取 tech_tree.ts 静态定义 + MVU 运行时状态
    ├── 派生字段：直接读 store.data.$穿越者存活数（`deriveAll` 自动生成）等；已解锁能力清单读 store.data.科技树.已解锁能力清单（非 $ 前缀，为 AI 维护数组）
    ├── 科技树工具：tech_tree_utils.ts 合并静态定义 + 研究槽运行时，提供限制校验与 DAG 数据
    └── 底部菜单显隐：核心 7 项常驻；财政/区划/人事按数据存在性追加
```

## 二、入口与骨架

### 2.1 index.html

严格遵循 `前端界面.mdc`：`<head>` 留空（webpack 注入），`<body>` 只放挂载点。

```html
<head></head>
<body>
  <div id="app"></div>
</body>
```

### 2.2 index.ts

参照 `示例/角色卡示例/界面/状态栏/index.ts`：等待 Mvu 框架与 stat_data 就绪后再挂载，避免空数据闪烁。

```ts
import { waitUntil } from 'async-wait-until';
import { createPinia } from 'pinia';
import { createApp } from 'vue';
import App from './App.vue';
import './global.css';

$(async () => {
  await waitGlobalInitialized('Mvu');
  await waitUntil(() => _.has(getVariables({ type: 'message' }), 'stat_data'));
  createApp(App).use(createPinia()).mount('#app');
});
```

### 2.3 store.ts

参照 `示例/角色卡示例/界面/状态栏/store.ts`，直接用 `defineMvuDataStore`。注意 import 路径：`store.ts` 在 `群友群穿/界面/状态栏/` 下，到 `src/群友群穿/` 需要 `../../..` 三层：

```ts
import { defineMvuDataStore } from '@util/mvu';
import { Schema } from '../../../src/群友群穿/mvu_schema';

export const useDataStore = defineMvuDataStore(
  Schema,
  { type: 'message', message_id: getCurrentMessageId() },
);
```

> **注意**：`defineMvuDataStore` 内部已实现 2s 轮询 + `watchIgnorable` 双向同步 + `schema.safeParse` 校验回写（见 `util/mvu.ts`）。组件只需 `useDataStore().data` 即可拿到响应式 `stat_data`，无需自行轮询。

> **导入路径易错点**：`store.ts` 在 `状态栏/` 目录下，而组件在 `状态栏/components/` 子目录中——组件比 store 深一层。因此组件中到 `src/群友群穿/` 的 import 需加一层 `..`（共 `../../../../src/群友群穿/...`），不可直接复制 store.ts 的路径。已知两次出错：`../../../src/群友群穿/constants` 从组件解析到 `群友群穿/src/群友群穿/constants`（不存在），正确应为 `../../../../src/群友群穿/constants`。

### 2.4 App.vue 主布局

采用手机案卷布局：顶部 `WorldSection` + `BaseVehicleBar` 是固定信息抬头；中间为当前案卷正文；底部 `TabNav` 是手机 APP 式菜单栏。需用 `useLocalStorage` 持久化当前案卷，并用横向滚动底部菜单适配窄屏。

```vue
<template>
  <div class="status-panel">
    <WorldSection />
    <BaseVehicleBar />

    <div class="dossier-body">
      <TechTreePanel v-if="active_tab === '科技'" />
      <OrganizationTree v-else-if="active_tab === '组织'" />
      <MilitaryPanel v-else-if="active_tab === '军事'" />
      <CharacterGrid v-else-if="active_tab === '角色'" />
      <ExternalForcesPanel v-else-if="active_tab === '外部'" />
      <IntelligencePanel v-else-if="active_tab === '情报'" />
      <EventQueue v-else-if="active_tab === '事件'" />
      <FinancePanel v-else-if="active_tab === '财政'" />
      <AdminDivisionPanel v-else-if="active_tab === '区划'" />
      <PersonnelPanel v-else-if="active_tab === '人事'" />
    </div>

    <TabNav v-model="active_tab" :tabs="visible_tabs" />
  </div>
</template>

<script setup lang="ts">
import { computed } from 'vue';
import { useLocalStorage } from '@vueuse/core';
import { useDataStore } from './store';
import WorldSection from './components/WorldSection.vue';
import BaseVehicleBar from './components/BaseVehicleBar.vue';
import TabNav from './components/TabNav.vue';
import TechTreePanel from './components/TechTreePanel.vue';
import OrganizationTree from './components/OrganizationTree.vue';
import MilitaryPanel from './components/MilitaryPanel.vue';
import CharacterGrid from './components/CharacterGrid.vue';
import ExternalForcesPanel from './components/ExternalForcesPanel.vue';
import IntelligencePanel from './components/IntelligencePanel.vue';
import EventQueue from './components/EventQueue.vue';
import FinancePanel from './components/FinancePanel.vue';
import AdminDivisionPanel from './components/AdminDivisionPanel.vue';
import PersonnelPanel from './components/PersonnelPanel.vue';

const store = useDataStore();

const core_tabs = [
  { id: '科技', label: '科技', icon: '🔬' },
  { id: '组织', label: '组织', icon: '🏛️' },
  { id: '军事', label: '军事', icon: '⚔️' },
  { id: '角色', label: '角色', icon: '👥' },
  { id: '外部', label: '外部', icon: '🌐' },
  { id: '情报', label: '情报', icon: '🕵️' },
  { id: '事件', label: '事件', icon: '⚠️' },
];

const optional_tabs = computed(() => {
  const tabs: { id: string; label: string; icon: string }[] = [];
  if (store.data.组织结构?.财政) tabs.push({ id: '财政', label: '财政', icon: '💰' });
  if (store.data.组织结构?.行政区划) tabs.push({ id: '区划', label: '区划', icon: '🗺️' });
  if (store.data.组织结构?.人事) tabs.push({ id: '人事', label: '人事', icon: '📋' });
  return tabs;
});

const visible_tabs = computed(() => [...core_tabs, ...optional_tabs.value]);

const active_tab = useLocalStorage<string>('群友群穿:status_bar:active_tab', '科技');
</script>

<style lang="scss" scoped>

  width: 100%;
  box-sizing: border-box;
  padding: 8px;
  color: var(--c-text);
  font-family: var(--font-ui);
  background: var(--c-bg-deep);

}

.dossier-body {
  padding: 14px 12px 86px;
}
</style>
```

> **可选菜单显隐策略**：参照 `后期扩展包施工指南.md` §2.1 的「按数据存在性触发」原则——底部菜单中 `财政/区划/人事` 判断 `store.data.组织结构.财政/行政区划/人事` 是否存在，初期 AI 不填则缺省、菜单项不显示；建政后 AI 按剧情自然补值，菜单项自然出现。**无开关、无锁定、无 schema 切换**。
> **交互约束**：底部菜单点击只切换案卷，不再允许点击当前项折叠为空。手机端应始终显示一个有效案卷页。

## 三、世界状态栏与基地车状态条

### 3.1 WorldSection.vue

参照 `示例/角色卡示例/界面/状态栏/components/WorldSection.vue` 的 meta-row + event-list 结构，但字段映射到群穿的「日期/时间/政治阶段/当前大事件/穿越天数/当前会议」。

```vue
<template>
  <div class="world-section">
    <div class="meta-row">
      <span>📅 {{ store.data.世界.当前日期 || '----.--.--' }}</span>
      <span>🕐 {{ store.data.世界.当前时间 || '--:--' }}</span>
      <span>⏳ 穿越第 {{ store.data.世界.穿越天数 }} 天</span>
    </div>
    <div class="phase-row">
      <span class="phase-tag">阶段：{{ store.data.世界.政治阶段 || '--' }}</span>
      <span v-if="store.data.世界.当前会议" class="meeting-tag" :class="{ active: isMeetingActive }">
        🏛️ {{ store.data.世界.当前会议 }}
      </span>
    </div>
    <div class="event-headline">
      <span class="event-label">当前大事件</span>
      <span class="event-text">{{ store.data.世界.当前大事件 || '暂无' }}</span>
    </div>
  </div>
</template>

<script setup lang="ts">
import { computed } from 'vue';
import { useDataStore } from '../store';

const store = useDataStore();

const isMeetingActive = computed(() => {
  const m = store.data.世界.当前会议;
  return m && m !== '无';
});
</script>
```

> **当前会议**：`v-if` 字段非空时显示，会议进行中（非「无」）高亮。初期 AI 不填则不显示，建政后自然出现。

### 3.2 BaseVehicleBar.vue（点击触发弹窗）

基地车状态条以图标 + 色阶徽标呈现反应堆/配电/外骨骼概要。

**交互改造**：整条 `vehicle-bar` 增加 `@click` 事件，点击后呼出 `BaseVehicleDetailModal`，展示基地车完整数据。点击区域加 `cursor: pointer` 视觉提示。

#### 使用的字段（从 store.data.基地车）

| 字段           | 显示位置                   |
| -------------- | -------------------------- |
| `当前位置`     | 条标题左：🚛 当涂县长江南岸 |
| `移动状态`     | 条标题右 · 分隔            |
| `反应堆状态`   | 标签徽标 ⚛️                 |
| `配电系统负载` | 标签徽标 ⚡%                |
| `外骨骼现状`   | 标签徽标 🦾                 |

#### 色阶映射（自由字符串 fallback）

```ts
/正常|充足|满功率|驻扎/ → var(--qyc-ok)
/紧张|降|过载|故障/  → var(--qyc-warn)
/危急|损毁|耗尽|失控/ → var(--qyc-danger)
未知取值 → var(--qyc-ink-dim)
```

#### 事件

- `@click` → 父组件通过 `ref` 控制 `BaseVehicleDetailModal` 显隐

色阶映射函数：正常→var(--qyc-ok)、降级/降功率/紧张→var(--qyc-warn)、故障/危急/损毁→var(--qyc-danger)。**注意反应堆状态/配电系统状态等均为自由字符串**，着色时需对未知取值做 fallback（默认色 var(--qyc-ink-dim)），参照 MVU 脚本施工指南的 OrganizationTree 注意事项。

## 三·五、设置功能：游玩中实时编辑面板数据

> **动机**：游玩过程中，AI 可能漏填或误填某些 stat_data 字段，GM/玩家需要能直接从状态栏界面修正数据，无需切回提示词或手动改变量。所有状态栏面板均应支持「只读浏览 ↔ 编辑模式」切换。

### 3.5.1 总体设计

每个数据展示面板增加「✎ 编辑」按钮（或等效切换入口），点击后同一区域内的展示文本变为可编辑控件（input / select / textarea）。编辑模式下的修改通过 `defineMvuDataStore` 的 `watchIgnorable` 自动双向同步回 `stat_data`，无需手动保存。

**设计原则**：

1. **编辑模式是局部状态**：每个面板独立控制 `editing` ref，互不影响。
2. **select + custom input 双轨**：已有参考列表的字段（如 `政治阶段`、`反应堆状态`）用 `<select>` 下拉 + 自定义输入文本框并行，玩家既可从参考列表快速选值，也可输入 AI 扩展的新值。
3. **z.record 可增删**：`核心设备状态` / `物资储备` 等 `z.record` 字段，编辑模式下显示可编辑键名 + 值字段 + 添加/删除按钮。
4. **直接操作 store.data**：编辑控件用 `v-model` 直接绑定 `store.data.模块.字段`，不经过中间 draft 状态——每次输入即双向同步。
5. **所有面板统一风格**：编辑按钮样式、表单控件尺寸、颜色方案在所有面板保持一致。

  - 底部有「＋ 添加设备/物资」按钮
  - 键名变更时同步更新 record 的 key（通过 watch 设备编辑键 / 物资编辑键 重建 record）

```ts

  基地车移动状态参考列表,
  反应堆功率等级参考列表,
  反应堆状态参考列表,

  核心设备状态参考列表,
  物资储备水平参考列表,
} from '../../../../src/群友群穿/constants';  // 组件在 components/ 下，需 4 层 ../
```

**z.record 编辑的关键逻辑**：

> **⚠️ 循环 watch 陷阱**：切勿使用 `watch(设备编辑键, ...)` 重建整个 record 来「同步」键名变更。这会导致：
>
> ```
> 核心设备列表 computed 变更 → watch 更新 设备编辑键 → watch 重建 v.value.核心设备状态 → 回到步骤 1（死循环）
> ```
>
> 正确方案：键名 input 用 `:value` + `@change` 事件，在 handler 中直接在底层 record 上 delete old key + set new key：

```ts
// 编辑模式下，维护一个与 核心设备列表 同步的 设备编辑键 数组
const 设备编辑键 = ref<string[]>([]);

watch(核心设备列表, (list) => {
  设备编辑键.value = list.map(([k]) => k);
}, { immediate: true });

function 添加核心设备() {
  const 设备状态 = v.value.核心设备状态 ?? {};
  const 新名 = `新设备 ${Object.keys(设备状态).length + 1}`;
  设备状态[新名] = { 状态: '正常', 备注: '' };
}

function 删除核心设备(idx: number) {
  const 旧键 = 设备编辑键.value[idx];
  if (旧键) delete v.value.核心设备状态[旧键];
}

// ✅ 键名变更：直接在底层 record 上重命名，不经过 record 重建
function 重命名核心设备(idx: number, 新键: string) {
  if (!新键) return;
  const 设备状态 = v.value.核心设备状态;
  if (!设备状态) return;
  const 旧键 = 设备编辑键.value[idx];
  if (旧键 === 新键 || !旧键) return;
  if (设备状态[新键]) return; // 键名冲突，跳过
  设备状态[新键] = 设备状态[旧键];
  delete 设备状态[旧键];
  设备编辑键.value[idx] = 新键;
}
```

模板中对应的 input 使用 `:value` + `@change` 而非 `v-model`：

```vue
<input
  :value="设备编辑键[idx]"
  @change="重命名核心设备(idx, ($event.target as HTMLInputElement).value)"
  type="text"
  class="form-input form-input--key"
  placeholder="设备名"
/>
```

物资储备的 `z.record` 编辑逻辑与此完全对称（`物资编辑键` / `重命名物资键` / `添加物资` / `删除物资`）。

| 1   | **前置依赖硬约束**：前置未完成不可添加                     | `节点是否可添加()` 检查 `前置科技.every(前 => 已完成节点.has(前))` |

| 3   | **槽位排他**：已在任一研究槽的节点不可重复添加             | `获取研究槽节点名()` 收集所有槽位节点，`节点是否可添加()` 检查排他 |
| 4   | **有效节点检查**：仅在 `tech_tree.ts` 定义的节点可添加     | `节点是否可添加()` 检查 `科技树定义[节点名]` 存在                  |
| 5   | **研究槽容量**：槽位数量不得超过当前上限                   | `主要/次要研究槽是否可添加()` 检查有效槽数 < 当前上限              |
| 6   | **科技树终局**：`大规模科研计划` 完成后锁定全部操作        | `科技树是否已完成()` 检查 `已完成节点.has('大规模科研计划')`       |

### 4.7 预计剩余天数

由于新 schema 使用 `研发进度(0-100%)` 而非 `已研发天数`，无法精确按天数计算。采用**保守估算**：

```ts
import { 预计剩余天数 } from './tech_tree_utils';

// 返回剩余等效天数
预计剩余天数('土法堆肥', 60);  // 返回 12（= (100-60)/100 × 30）
```

公式：`剩余天数 = ceil((100 - 研发进度) / 100 × 研发总耗时_天)`

在卡片和槽位中展示：`预计剩余: {剩余天数} 天`

> **不区分主/次速率**：旧版三档时间预估（100%/30%/5%）是基于旧版「按方向速率累加已研发天数」的设计。新版使用 `研发进度(0-100)` 直接由 AI 按剧情判断填写，前台无法区分速率。若需要精确时间线，可在兜底脚本中按启动日期+日历流逝天数计算应完成比例并输出到 `已解锁能力清单` 条目的日期后缀中。

### 4.9 科技树终局界面（大规模科研计划完成后）

```ts
function 建机构树(机构表: Record<string, 机构Node>) {
  const 树: Record<string, 机构Node & { 子机构: 机构名[] }> = {};
  // 第一遍：初始化
  for (const [名, 机构] of Object.entries(机构表)) {
    树[名] = { ...机构, 子机构: [] };
  }
  // 第二遍：挂载子机构
  for (const [名, 机构] of Object.entries(树)) {
    if (机构.上级机构 && 树[机构.上级机构]) {
      树[机构.上级机构].子机构.push(名);
    }
  }
  // 顶层机构 = 上级机构为 '' 的节点
  return Object.entries(树).filter(([, n]) => n.上级机构 === '');
}
```

```ts
function 推导顶层机构(角色名: string, 机构表: Record<string, 机构Node>): string {
  for (const [机构名, 机构] of Object.entries(机构表)) {
    if ([机构.一把手, 机构.二把手, ...机构.副职, ...机构.成员].includes(角色名)) {
      // 沿 上级机构 链上溯至顶层
      let 当前 = 机构名;
      while (机构表[当前]?.上级机构) 当前 = 机构表[当前].上级机构;
      return 当前;
    }
  }
  return '无所属';
}
```

## 十、iframe 适配硬性规范

遵循 `.cursor/rules/前端界面.mdc`：

1. **禁用 `vh`**：高度调整用 `width` + `aspect-ratio`，或让内容自然撑高（但需避免 `min-height` / `overflow: auto` 强制撑高父容器）
2. **94 人网格需虚拟滚动或分页**，避免撑高父容器
3. **优先 tailwindcss**，复杂样式用 `<style lang="scss" scoped>`
4. **页面必须有外部支撑**，主体内容不能用 `position: absolute` 脱离文档流（`InstitutionDetailModal` 的 `position: fixed` 是例外，因为它用 Teleport 挂载到 body 且有遮罩）
5. **页面整体应适配容器宽度，不产生横向滚动条**；TabNav 在窄屏需横向滚动（`overflow-x: auto` + 隐藏滚动条）
6. **`<img>` 标签禁止 `src=""` 占位**，会导致 webpack 打包错误

## 十三、关键注意事项

### 13.1 数据流单向原则

- 状态栏**只读** `store.data`，除「研究方向选择」等少数玩家/GM 策略决策字段外，**不直接编辑 stat_data**
- 玩家策略决策（如主要研究方向）通过 `defineMvuDataStore` 的 `watchIgnorable` 双向同步写回，由兜底脚本在下一轮 `VARIABLE_UPDATE_ENDED` 时按新值累加
- AI 的叙事更新走 MVU 常规流程（`变量更新规则.yaml` + 兜底校验），状态栏只负责展示

### 13.2 自由字符串着色 fallback

`运作状态` / `反应堆状态` / `人身安全` / `财政状况` 等均为自由字符串（AI 可按剧情扩展），着色时必须对未知取值做 fallback：

```ts
function 状态色阶(值: string, 映射: Record<string, string>): string {
  return 映射[值] ?? 'var(--c-text-dim)'; // 未知取值用默认色
}
```

### 13.3 静态结构 join

科技树面板必须 join `tech_tree.ts` 静态定义 + MVU 运行时状态，**不能只读 MVU**（MVU 只存已研发天数/启动时间/完成时间，不含前置依赖/研发总耗时/解锁能力）。新增节点流程：先改 `科技树DAG指南.md` → 同步改 `tech_tree.ts` → 无需改状态栏。

### 13.4 编辑模式强制原则

> **黄金法则**：状态栏中**每一个展示数据的面板和弹窗**都必须提供编辑模式。原因是：游玩中 AI 时常漏填或误填字段，GM 必须能就地修正而不中断叙事。
>
> 唯一的合法例外是纯导航/布局组件（`TabNav`、`BaseVehicleBar` 状态条自身）。

所有编辑模式遵循 §三·五 的统一样式与交互模式：`edit-btn` 按钮 + `form-input`/`form-select` 控件 + `select+custom 双轨` + `z.record 可增删`。

### 13.5 性能

- 305 节点科技树：默认按分支折叠，只展开当前主要方向所在分支
- 94 人群像：分页（每页 24 人）或虚拟滚动
- `defineMvuDataStore` 已内置 2s 轮询 + `_.isEqual` 去重，组件无需自行轮询
- 避免在 `computed` 中做重计算（如拓扑分层），可用 `watchEffect` + `ref` 缓存

### 13.6 与后期扩展包的边界

本指南已合并 `后期扩展包施工指南.md` §6 的全部状态栏扩展内容（`InstitutionDetailModal` / `FinancePanel` / `AdminDivisionPanel` / `PersonnelPanel` / 三权指针 / 当前会议）。**后期扩展包施工指南已完成其使命**，后续状态栏施工以本指南为准。
