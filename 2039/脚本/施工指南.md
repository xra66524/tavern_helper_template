# 2039 MVU 变量系统 —— 施工指南

## 一、概述

本指南针对 2039（Arma3世界观 × 邦邦群像剧）角色卡，规划其 MVU 变量（`mvu_data`）的完整施工流程。

Schema 由 **7个顶层模块**、**13个子Schema**、**约120个字段** 构成，围绕"权力运作的因果链"组织：

```
事件发生 → 政治决策 → 政策出台 → 影响经济/社会 → 反馈到支持度/舆论 → 触发新事件
```

其中 `日本执政` 是核心枢纽模块，内含 `本届内阁`、`阁僚`、`政策`、`选举进程`、`事务次官会议`、`各省厅`、`主要政治势力` 共7个子Schema，占全Schema约60%的字段量。

### 核心原则

1. **变量追踪"会变化的东西"，而非静态设定**——世界书已提供的设定，不需要在变量中重复。
2. **采用"渐进式"施工**——先搭骨架（核心模块），再填充细节（扩展模块），最后打磨（派生计算 + 优化）。
3. **保持Schema的向下兼容性**——每个阶段新增字段不破坏已有数据。
4. **充分利用 `.transform()` 做派生计算**——减少AI需要直接控制的原始数值，让Schema自动推导出状态标签。
5. **分片更新，降低AI负担**——每轮只更新与当前剧情相关的模块，而非全量更新。

### 分片更新机制

> **问题**：当前Schema包含7个顶层模块、13个子Schema、约120个字段。如果AI每轮都要更新全部变量，token消耗会爆炸，且AI难以同时精确控制这么多字段。
>
> **解决方案**：利用MVU框架的`_.set()`命令机制，每轮只更新与当前剧情相关的**分片**，未更新的字段自动保留原值。

#### 原理

MVU的变量更新流程：

```
AI回复中嵌入 _.set('路径', 新值) 命令
     │
     ▼
Mvu.parseMessage() 解析命令，只修改指定路径
     │
     ▼
Schema.safeParse() 清洗/裁剪/派生（未修改的字段保留原值）
     │
     ▼
写入 stat_data
```

**关键特性**：`_.set()` 是路径级别的更新，不需要提供完整对象。AI只需更新变化的字段。

#### 分片策略

将7个顶层模块按**剧情相关性**分为3个更新优先级：

| 优先级   | 模块         | 更新频率        | 说明                                        |
| -------- | ------------ | --------------- | ------------------------------------------- |
| **必更** | 世界         | 每轮            | 时间推进是剧情的基础                        |
| **必更** | 事件队列     | 每轮            | 事件驱动是剧情引擎                          |
| **高优** | 日本执政     | 触及政治时      | 核心模块（含7个子Schema），但不是每轮都涉及 |
| **高优** | 舆论场       | 触及舆论时      | 与选举/政策强相关                           |
| **按需** | 日本国内情势 | 触及经济/社会时 | 经济社会变化较慢，不需要每轮更新            |
| **按需** | 国际势力动态 | 触及外交时      | 外部势力行动不频繁                          |
| **按需** | 各角色情况   | 触及角色时      | 只更新出场的角色                            |

#### 实现方式

**方式一：通过正则/脚本控制AI的更新范围**

在系统提示或正则中，根据当前`世界.政治阶段`和`事件队列.当前大事件`，动态生成"本轮应更新的模块"提示：

```
当前政治阶段：特别大选初选期
本轮建议更新模块：世界 + 事件队列 + 日本执政.选举进程 + 舆论场
本轮可忽略模块：日本国内情势 + 国际势力动态 + 各角色情况
```

**方式二：通过`VARIABLE_UPDATE_ENDED`事件做后处理**

在脚本中监听`Mvu.events.VARIABLE_UPDATE_ENDED`，根据已更新的字段自动推导关联字段：

```ts
eventOn(Mvu.events.VARIABLE_UPDATE_ENDED, (variables, variables_before) => {
  const data = variables.stat_data;

  // 如果选举进程变化了，自动推进政治阶段
  if (!_.isEqual(data.日本执政?.选举进程, variables_before.stat_data.日本执政?.选举进程)) {
    // 自动推导 世界.政治阶段
    const 阶段 = derive政治阶段(data.日本执政.选举进程);
    _.set(data, '世界.政治阶段', 阶段);
  }

  // 如果政策变化了，自动影响经济民生
  if (!_.isEqual(data.日本执政?.当前政策, variables_before.stat_data.日本执政?.当前政策)) {
    // 自动推导经济影响
    derive经济影响(data);
  }

  // 如果事件发生了，自动影响舆论场
  if (!_.isEqual(data.事件队列?.近期已发生事件, variables_before.stat_data.事件队列?.近期已发生事件)) {
    derive舆论影响(data);
  }
});
```

**方式三：模块级Schema拆分（可选，更激进）**

将7个模块拆分为独立的Schema，各自注册到不同的变量路径：

```ts
// 方案A：单一Schema（当前方案，简单但更新粒度粗）
registerMvuSchema(完整Schema);

// 方案B：拆分Schema（更灵活但管理复杂）
registerMvuSchema(世界Schema);        // stat_data.世界
registerMvuSchema(日本执政Schema);     // stat_data.日本执政
registerMvuSchema(国内情势Schema);     // stat_data.日本国内情势
// ... 各自独立更新
```

> **推荐**：初期采用方案A（单一Schema），通过方式一+方式二实现分片更新。如果后续发现单一Schema的token消耗仍然过大，再考虑方案B。

#### 分片更新的AI提示模板

在正则或世界书中，为AI提供"本轮更新指引"：

```
【变量更新指引】
当前阶段：{世界.政治阶段}
当前焦点：{事件队列.当前大事件}

本轮必须更新：
- 世界.当前日期 / 世界.当前时间
- 事件队列.近期已发生事件（如有新事件）

本轮按需更新（仅更新与剧情相关的部分）：
- 日本执政：仅更新与当前剧情直接相关的子字段（如选举进程/阁僚状态/政策）
- 舆论场：仅当涉及媒体/网络/民意时更新
- 各角色情况：仅更新本轮出场的角色

本轮通常不需要更新：
- 日本国内情势 / 国际势力动态（除非发生重大变化）
```

#### 分片更新的数据一致性保障

分片更新的风险是"更新了A但忘了联动B"。通过以下机制保障一致性：

1. **`VARIABLE_UPDATE_ENDED`后处理**：脚本自动推导模块间联动
2. **`.transform()`派生计算**：Schema自身保证派生字段的一致性
3. **2秒轮询同步**：`defineMvuDataStore`内置的轮询机制保证外部修改能同步到UI

## 二、施工阶段

### 阶段一：基础设施 + 核心骨架（第1轮）

**目标**：搭建完整的Schema结构框架，所有字段以最小可用形式存在，保证可注册运行。

#### Step 1.1 — 项目文件结构

创建 `src/2039_mvu/` 目录，包含以下文件：

```
src/2039_mvu/
├── constants.ts      ← 枚举值与常量（政治阶段7种、选举模式3种、省厅15+、势力6+、角色按需）
├── transforms.ts     ← 派生计算函数（$执政合法性、$政府控制力、$整体危机等级、$当前剧情焦点、$外部压力指数）
├── mvu_schema.ts     ← 主Schema（7个顶层模块 + 13个子Schema，约120个字段）
├── index.ts          ← registerMvuSchema 注册 + VARIABLE_UPDATE_ENDED 模块间联动
└── types.ts          ← z.output<typeof Schema> 类型导出
```

**各文件职责详解**：

- **`constants.ts`** — 所有枚举值和常量列表，供Schema引用：
  - `政治阶段列表`：遇刺潮末期 / 看守停摆期 / 正常选举期 / 特别大选报名期 / 特别大选初选期 / 特别大选指名期 / 新内阁执政期
  - `选举模式列表`：正常首相指名 / 2039特别大选 / 无
  - `省厅列表`：外务省 / 防卫省 / 经产省 / 农水省 / 财务省 / 内阁官房 / 警察厅 / 国交省 / 法务省 / 文科省 / 厚劳省 / 总务省 / 环境省 / 公安委 / 内阁府（含下属局厅映射）
  - `势力列表`：CSAT / 美国·CIA / 朝总联 / NATO / 俄罗斯·GRU / 西方情报联盟
  - `角色列表`：按剧情需要动态维护，初始为空

- **`transforms.ts`** — 可复用的 `.transform()` 派生计算函数：
  - `derive执政合法性(首相状态, 选举进程, 主流媒体立场, 内阁稳定度)` → `$执政合法性`
  - `derive政府控制力(事务次官会议权力等级, 各省厅内阁掌控度均值, 内阁稳定度)` → `$政府控制力`
  - `derive整体危机等级(治安等级, 通胀等级, 政治阶段, 粮食供应, 潜在危机)` → `$整体危机等级`
  - `derive当前剧情焦点(政治阶段, 大事件, 选举进程, 各角色情况)` → `$当前剧情焦点`
  - `derive外部压力指数(国际势力动态)` → `$外部压力指数`

- **`mvu_schema.ts`** — 主Schema定义，包含7个顶层模块和13个子Schema：
  - 顶层：`世界` / `日本执政` / `日本国内情势` / `国际势力动态` / `舆论场` / `各角色情况` / `事件队列`
  - 子Schema：`本届内阁` / `阁僚` / `政策` / `选举进程` / `事务次官会议` / `各省厅` / `政治势力` / `势力内部派系` / `已决议事项` / `候选人` / `区域分化` / `社会群体` / `角色情况`

- **`index.ts`** — 入口文件：
  - `registerMvuSchema(Schema)` 注册
  - `VARIABLE_UPDATE_ENDED` 事件监听：选举进程→政治阶段联动、政策→经济民生联动、事件→舆论联动、阁僚↔角色安全状况同步

- **`types.ts`** — 类型导出：`export type MvuData = z.output<typeof Schema>`

#### Step 1.2 — 顶层结构

> **设计理念**：2039是一张"模拟政府执政"的卡，顶层结构必须反映**权力运作的因果链**：
> 事件发生 → 政治决策 → 政策出台 → 影响经济/社会 → 反馈到支持度/舆论 → 触发新事件。
> 因此模块之间不是平铺的，而是围绕"执政"这个核心形成嵌套与联动。

```
Schema = z.object({
  世界: 世界Schema,                     // 时间轴 + 阶段标记（4字段）
  日本执政: 日本执政Schema,             // 核心：内阁 + 政策 + 选举（7个子Schema，~70字段）
  日本国内情势: 日本国内情势Schema,     // 经济民生 + 社会秩序的合并仪表盘（~25字段，10子分组）
  国际势力动态: 国际势力动态Schema,     // 外部压力源（z.record，每势力8字段）
  舆论场: 舆论场Schema,                 // 执政合法性的"晴雨表"（3跨群体指标 + z.record按群体索引）
  各角色情况: 各角色情况Schema,         // 群像追踪（z.record，每角色8维度~15字段）
  事件队列: 事件队列Schema,             // 事件驱动引擎（近期事件 + 潜在危机）
})
```

**各模块定位说明**：

| 模块         | 定位         | 核心问题                                 | 子Schema数 | 字段量     |
| ------------ | ------------ | ---------------------------------------- | ---------- | ---------- |
| 世界         | 时间轴       | 现在是什么时候？处于什么阶段？           | 0          | ~4         |
| 日本执政     | **权力中枢** | 谁在执政？推行什么政策？选举到哪一步了？ | 7          | ~70        |
| 日本国内情势 | 执政效果     | 老百姓过得怎么样？社会安定吗？           | 0          | ~25        |
| 国际势力动态 | 外部压力     | 谁在施压？在争夺什么？                   | 0          | 8×势力数   |
| 舆论场       | 合法性来源   | 民众怎么看？互联网上在吵什么？           | 0          | 3+6×群体数 |
| 各角色情况   | 群像追踪     | 每个角色现在在做什么？                   | 0          | ~15×角色数 |
| 事件队列     | 剧情引擎     | 刚发生了什么？接下来可能发生什么？       | 0          | ~4         |

**日本执政模块的内部结构**（核心枢纽，占全Schema约60%字段量）：

```
日本执政
├── 首相状态 / 首相姓名                    ← 顶层字段
├── 本届内阁Schema                         ← 内阁成员 + 执政联盟
│   └── 主要阁僚: z.record(职位名, 阁僚Schema)  ← 双线追踪（政治能力 + 少女视角）
├── 选举进程Schema                         ← 支持正常指名 / 特别大选两种模式
│   └── 候选人排名: z.array(候选人Schema)
├── 事务次官会议Schema                     ← 日本实质最高权力中心
│   ├── 内部派系: z.record(派系名, 势力内部派系Schema)
│   └── 已决议事项: z.array(已决议事项Schema)
├── 主要政治势力: z.record(势力名, 政治势力Schema)
├── 各省厅: z.record(省厅名, 各省厅Schema)  ← 双重掌控度设计
└── 当前政策: z.record(政策名, 政策Schema)  ← 政策完整生命周期追踪
```

#### Step 1.3 — 各模块最小骨架

**世界 Schema**（时间轴 + 阶段标记）

```ts
世界: z.object({
  当前日期: z.string(),
  当前时间: z.string(),
  政治阶段: z.enum([
    '遇刺潮末期',          // 2038.11 高市早苗遇刺后
    '看守停摆期',          // 无人敢竞选首相
    '正常选举期',          // 议会内进行首相指名选举
    '特别大选报名期',      // 2039特殊机制：互联网报名
    '特别大选初选期',      // 2039特殊机制：互联网初选
    '特别大选指名期',      // 2039特殊机制：议会最终指名投票
    '新内阁执政期',        // 选出新任首相后
  ]),
  选举模式: z.enum(['正常首相指名', '2039特别大选', '无']),
  当前大事件: z.string(),
})
```

**日本执政 Schema**（权力中枢 —— 本卡的核心模块）

```ts
日本执政: z.object({
  首相状态: z.enum(['空位', '看守', '新任', '在任']),
  首相姓名: z.string(),
  本届内阁: 本届内阁Schema,           // 内阁成员 + 执政联盟
  选举进程: 选举进程Schema,           // 支持「正常首相指名」和「2039特别大选」两种模式
  事务次官会议: 事务次官会议Schema,   // 官僚系统的实际权力
  主要政治势力: z.record(势力名, 政治势力Schema),
  各省厅: z.record(省厅名, 省厅Schema),
  当前政策: z.record(政策名, 政策Schema),  // 🆕 正在推行的政策/法案
})
```

**本届内阁 Schema**（🆕 追踪执政团队 —— 核心设计：少女执政的荒诞与危险）

> **设计理念**：2039的戏剧核心是"不通政治的少女被推上权力巅峰"。阁僚Schema必须同时追踪两条线：
>
> - **表面线**：官方职位、政治立场、忠诚度（传统政治维度）
> - **实质线**：她懂不懂？谁在操控她？她到底在不在干活？她心里在想什么？
> 这两条线的落差越大，戏剧性越强。

```ts
本届内阁: z.object({
  届数: z.coerce.number(),
  首相政治立场: z.enum(['极右翼', '右翼', '中间偏右', '中间', '中间偏左', '左翼', '不确定']),
  执政联盟: z.array(z.string().describe('联盟势力名')),
  内阁稳定度: z.coerce.number().transform(v => _.clamp(v, 0, 100)),
  主要阁僚: z.record(z.string().describe('职位名，如「外务大臣」「防卫大臣」'), 阁僚Schema),
})
```

**阁僚 Schema**（核心：追踪"人"而非"位"）

```ts
阁僚: z.object({
  // === 身份信息 ===
  姓名: z.string(),
  年龄: z.coerce.number(),
  原身份: z.string().describe('就任前的身份，如「花咲川高三学生」「前自民党议员」「朝总联干部」'),

  // === 政治能力（核心落差维度） ===
  政治立场: z.string().describe('如「极右翼」「中间」「不懂政治」「反CSAT」'),
  对职务理解度: z.enum(['完全不懂', '一知半解', '勉强理解', '熟练', '专家']),
  实际履职状态: z.enum(['正常履职', '敷衍了事', '被架空', '完全摆烂', '恐慌中', '逃避职责']),

  // === 操控与被操控（双向权力关系） ===
  幕后操控者: z.string().describe('试图操控该职位的人或势力，如「事务次官会议」「CIA」「朝总联」「无（自主决策）」'),
  操控程度: z.coerce.number().transform(v => _.clamp(v, 0, 100)).describe('0=完全自主，100=纯傀儡'),
```

**双向权力关系说明**：

阁僚与官僚/利益集团之间不是简单的"操控vs被操控"，而是一个**动态博弈光谱**：

```
完全傀儡 ◄──────────────────────────────────────► 完全自主
   │                                                    │
   │  操控程度: 100          操控程度: 0                 │
   │  反向影响力: 0          反向影响力: 100              │
   │                                                    │
   │  ←── 可能的演变路径 ──→                            │
   │                                                    │
   │  阶段1: 完全服从（无争斗）                          │
   │  阶段2: 暗中摩擦（开始有不同意见）                   │
   │  阶段3: 公开对抗 / 拉拢分化                         │
   │  阶段4: 逐步夺权                                    │
   │  阶段5: 已掌控局面                                  │
```

**典型演变场景**：

| 场景                   | 操控程度 | 反向影响力 | 争斗状态   | 官僚应对           |
| ---------------------- | -------- | ---------- | ---------- | ------------------ |
| 刚上任的邦邦少女       | 90       | 5          | 无争斗     | 无（无需应对）     |
| 她发现官僚在骗她       | 70       | 20         | 暗中摩擦   | 阳奉阴违           |
| 她用偶像人气发动舆论   | 40       | 50         | 公开对抗   | 向媒体泄露负面信息 |
| 她拉拢了官僚中的改革派 | 20       | 70         | 拉拢分化   | 妥协让步           |
| 她成功掌控了省厅       | 5        | 90         | 已掌控局面 | 被迫服从           |

**影响力来源的多样性**（为什么一个不懂政治的少女也能反向影响官僚）：

- **偶像人气**：社交媒体一条发言就能引发舆论风暴，官僚不敢轻视
- **"外行"优势**：不受官僚惯例束缚，可能做出官僚不敢做的决策
- **乐队/同伴网络**：其他阁僚是她的乐队成员，形成跨省厅联盟
- **民众同情**：少女被推上危险位置的形象，天然获得民众支持
- **外部势力拉拢**：CIA/朝总联可能主动接触她，她可以反过来利用
- **无知者无畏**：不懂政治反而不会恐惧，可能做出大胆举动

**设计要点说明**：

| 维度         | 传统政客   | 邦邦少女        | Schema如何体现                                |
| ------------ | ---------- | --------------- | --------------------------------------------- |
| 对职务理解   | 熟练       | 完全不懂        | `对职务理解度`                                |
| 谁在真正干活 | 自己       | 事务次官        | `幕后操控者` + `操控程度`                     |
| 心里在想什么 | 权力博弈   | 下周的Live      | `更关心的事`                                  |
| 被暗杀风险   | 有心理准备 | 毫无概念        | `暗杀威胁等级` + `对当前职位的态度`           |
| 实际在干什么 | 开会决策   | 敷衍/恐慌/逃避  | `实际履职状态`                                |
| 与官僚的关系 | 博弈/共谋  | 被操控→可能反抗 | `与官僚体系的争斗状态` + `对官僚体系的影响力` |
| 反向影响力   | 权力基础   | 偶像人气/同情   | `影响力来源` + `官僚体系的应对`               |

**政策 Schema**（🆕 政府执政的核心产出 —— 追踪政策的完整生命周期）

> **设计理念**：政策不是凭空产生的。在2039中，每项政策都应该能追溯到：
>
> - 是谁提出的？（阁僚本人？事务次官？外部势力？）
> - 提出者懂自己在推行什么吗？（少女签字的荒诞性）
> - 政策实际产生了什么效果？（预期vs现实的反差）

**各省厅 Schema**（🏛️ 国家机器的"骨骼"—— 省厅是政策执行的基本单元）

> **设计定位**：省厅Schema不追踪高频变化（具体政策细节由 `当前政策` 模块处理），而是提供**省厅层面的宏观状态快照**。使用 `z.record()` 意味着**省厅条目可以随剧情动态新增或删除**（如内阁改组拆分省厅、紧急事态下新设临时机构等），初始列表由 `constants.ts` 提供基准。
>
> **双重掌控度设计**：2039年日本政体的核心矛盾之一，是少女阁僚（名义上的长官）与事务次官（实质掌控者）之间的权力拉锯。因此省厅Schema的掌控度必须分两条线追踪：
>
> - **事务次官会议掌控度**：该省厅的实际事务运作被官僚体系控制到什么程度？
> - **内阁掌控度**：该省厅的阁僚（大臣）能实际驱动这个部门吗？
>
> 两者可以独立变化：一个省厅可能事务次官掌控度90（官僚铁板一块），内阁掌控度10（大臣被架空）；也可能事务次官掌控度30（官僚内部分裂），内阁掌控度70（大臣意外地有手腕）。
>
> **下属局厅列出**：为防止AI在叙述中混入不存在的部门（如把中国的"发改委""公安部"写进日本省厅），每个省厅标注其关键下属机构，供AI和用户参考。
>
> **更新频率**：仅在省厅层面发生重大变化时更新（如被事务次官会议某个派系"攻陷"、运作状态降级、政治倾向转向、或内阁掌控度显著变化）。

**国际势力动态 Schema**（按势力/组织索引，而非按国家）

> **设计理念**：2039年的日本处于多个外部势力的角力场中央。本模块只追踪**会变化的东西**——对日策略动向、在日行动能力变化、内部博弈演变。势力全称、组织架构、历史沿革等静态设定由世界书提供，不在变量中重复。
>
> **核心原则**：每个势力用 `z.string()` 自由描述而非穷举枚举，让AI用一句话概括当前态势，而非逐字段填表。

```ts
国际势力动态: z.record(z.string().describe('势力名，如「CSAT」「美国/CIA」「朝总联」「NATO」「俄罗斯/GRU」「西方情报联盟」'), z.object({
  // === 对日策略（最核心：这个势力现在对日本在干什么？） ===
  对日策略倾向: z.enum(['积极扩张', '维持现状', '收缩撤出', '观望待机', '暗中渗透', '对抗遏制']),
  对日当前目标: z.string().describe('当前最想在日本达成的目标，如「通过CSDC将日本纳入经济圈」「完成驻日美军撤军」'),
  对日本政府态度: z.enum(['承认并合作', '观望', '暗中操控', '敌对', '无视']),
  对特别大选态度: z.enum(['积极干预', '暗中扶持', '观望', '试图破坏', '不关心']),

  // === 在日行动能力（这个势力在日本还能干多少事？） ===
  在日行动能力: z.enum(['无', '极低', '低', '中等', '高', '极高']),
  在日情报网络: z.enum(['无', '严重受损', '受损但运作', '正常运作', '扩张中']),
  近期在日行动: z.string().describe('近期在日最重要的1-2个行动及结果，如「推动粮食进口令但遭农协起诉违宪」'),

  // === 内部博弈（这个势力内部在对日问题上有没有分歧？） ===
  内部团结度: z.coerce.number().transform(v => _.clamp(v, 0, 100)),
  内部矛盾: z.string().describe('对日问题上的内部分歧，如「白宫收缩派 vs 五角大楼坚守派」「中俄能源定价权之争」'),

}))
```

**舆论场 Schema**（🆕 执政合法性的"晴雨表"）

> **设计理念**：2039年的日本社会高度分裂——都市青年在Live House逃避现实，农民因走私粮倾销愤怒，港口城市居民依赖CSAT贸易维生，冲绳怀念美国时代，沼津武装反CSAT。不同群体对同一事件的反应截然不同，因此舆论场不能只追踪"全民平均情绪"，必须按社会群体分别追踪。
>
> **设计模式**：顶层保留跨群体指标（社交媒体热点、网络风向、主流媒体立场），用 `z.record()` 按社会群体索引各自的舆论状态。只追踪当前活跃的群体（通常3-6个），随剧情发展可增删。


**各角色情况 Schema**（群像追踪）

> **设计理念**：2039是一张"群像剧"卡——没有单一主角，数十个角色在崩溃的日本各自挣扎。不是所有角色都是政治人物，但**所有角色都被政治影响**。这个模块需要追踪从"完全未卷入"到"已是阁僚"的完整光谱，同时保持对"人物日常"的关注——因为正是政治与日常的反差构成了2039的戏剧张力。
>
> **核心原则**：
>
> - 用 `z.record()` 按姓名索引，只追踪当前活跃的角色（通常5-15人），不需要为所有角色维护条目
> - **政治卷入度**是本模块的核心维度——它决定了一个角色在2039叙事中的位置
> - 保持对"人物视角"的追踪——角色真正关心的事往往不是政治，而是生计、Live、同伴、家人
> - 安全状况和经济状况是2039特有的硬指标——粮食危机和暗杀潮直接影响每个角色

**与阁僚 Schema 的关系**：

> 当某个角色的 `政治卷入度` 为「深度卷入」且 `担任公职` 不为「无」时，该角色的 `各角色情况` 条目与 `日本执政.本届内阁.主要阁僚` 中的对应条目**存在信息重叠但视角不同**：
>
> | 维度       | 阁僚 Schema（政治视角）           | 各角色情况（人物视角）     |
> | ---------- | --------------------------------- | -------------------------- |
> | 关注点     | 政治能力、权力关系、政策推行      | 日常生活、心理、人际关系   |
> | 操控程度   | ✅ 详细追踪                        | ❌ 不追踪（在阁僚Schema中） |

### 阶段二：字段精细化（第2轮）

**目标**：逐个模块填充细节字段，使Schema能够覆盖世界书中的核心动态要素。

#### Step 2.1 — 日本执政展开

**选举进程**（区分两种选举模式）

**正常首相指名流程**：

- 执政党党首选举（自民党总裁选举）
- 在野党协商与联合提名
- 众议院首相指名投票
- 参议院首相指名投票（两院不一致时以众议院为准）
- 如无人愿意出任首相→陷入僵局→可能触发特别大选切换

**2039特别大选流程**（由《国家紧急状态规定》触发）：

- 互联网报名（任意12岁以上日本公民可登记）
- 互联网初选（全民网络投票筛选候选人）
- 议会最终指名投票（对通过初选的候选人进行表决）
- 每隔一定时间重开一轮，直至选出愿意就任者

**两种模式的切换条件**：

- 正常指名投票连续失败 → 可切换为特别大选
- 特别大选选出首相后 → 恢复正常执政，选举模式复位

**可复用字段**（两种模式共用）：

- 候选人排名的结构（姓名/背景/票数/威胁等级/外部支持）
- 选举热度
- 备注（特殊情况说明）

**模式特有字段**：

- 正常模式：执政党推荐候选人 / 在野党联合候选人 / 议会内各党派议席分布
- 特别大选：互联网直选排名TOP恶搞名单 / 青少年参选人比例 / 外部势力操纵痕迹

**本届内阁**（🆕 阶段一已建骨架，此处展开）

- 阁僚的"双线追踪"细化：
  - 表面线：官方立场、公开表态、议会发言
  - 实质线：谁在操控、她实际在干什么、她心里在想什么
- 阁僚间的"少女关系"如何影响政治决策：
  - 乐队成员在同一内阁→私下讨论国事变成乐队排练间隙闲聊
  - 不同乐队的阁僚→跨乐队关系影响派系联盟
  - 阁僚与外部角色的关系→如"外务大臣的闺蜜是朝总联相关人士"
- 内阁与事务次官会议的权力拉锯：
  - 哪些阁僚被完全架空（操控度90+）
  - 哪些阁僚意外地开始自主决策（操控度下降）
  - 事务次官会议对"失控阁僚"的应对

**事务次官会议**

- 内部派系争斗追踪：各派系实力权重变化、派系间的合纵连横
- 与内阁的权力拉锯：
  - 哪些阁僚被事务次官会议操控（对方僚Schema的联动）
  - 哪些阁僚正在反抗（与阁僚Schema.与官僚体系的争斗状态联动）
  - 事务次官会议对"失控阁僚"的应对策略
- 与议会的关系：预算审议、立法提案权、国政调查权的博弈
- 与外部势力的暗中博弈：
  - CSAT渗透渠道（通过哪个省厅的哪个派系）
  - CIA残余网络（谁在保持联系、交换什么筹码）
  - 朝总联的灰色合作（走私默许的边界在哪）
- 与独立机构的冲突：地检署是否在调查事务次官会议？警察厅是否配合？
- 决策机制的变化：从"全体一致"到"强权主导"的演变
- 事务次官会议自身的合法性危机：媒体曝光、国会质询、民众不信任

**当前政策**（🆕 核心新增）

- 追踪每项政策的完整生命周期：来源→推行→效果→副作用
- 政策与阁僚的联动：谁提出的？提出者懂不懂内容？谁在背锅？
- 预期vs现实的反差追踪：政策宣称的效果 vs 实际产生的效果
- 政策类型区分：内阁提案 / 官僚推动 / 外部势力施压 / 紧急应对 / 象征性政策
- 政策间的连锁反应：一项政策的副作用可能催生下一项政策
- 政策与经济民生/社会秩序的联动（如：开放港口→影响汇率→影响通胀→引发右翼反弹）

**政治势力扩展**
从目前的"光谱+诉求+支持度"扩展到：

- 势力类型：政党 / 武装团体 / 官僚系统 / 灰色势力 / 地方豪强
- 对CSAT态度 / 对美态度
- 当前策略（观望/对抗/渗透/合作）
- 武装力量等级（针对非政党势力）

**各省厅**（阶段一已建骨架）

省厅 Schema 已在阶段一定义完毕，包含 `运作状态` / `掌控度` / `对CSAT态度` / `对美态度` / `倾向派系` / `当前焦点` 共6个字段。阶段二的重点是在 `constants.ts` 中确立追踪的省厅名单（建议优先聚焦外务省/防卫省/经产省/农林水产省/警察厅等核心省厅，其余视剧情按需补充）。

#### Step 2.2 — 国际势力动态展开

> 阶段一已建立精简的国际势力 Schema（每势力8个字段）。阶段二的重点是**完善 `特有关注` 字段的填写规范**，确保AI知道每个势力最该追踪什么动态，而非增加更多字段。

**西方情报联盟** — `特有关注`应追踪：CIA恢复进度、五眼裂痕演变、对CSAT情报战攻防态势

#### Step 2.3 — 国内情势展开（经济民生 + 社会秩序合并）

> 阶段一已建立精简的国内情势 Schema（~25个字段，10个子分组）。阶段二的重点是**完善各 `z.string()` 描述字段的填写规范**，确保AI知道该写什么细节。

- 金融因果链的演变追踪：`日元贬值趋势`→`货币信任度`→`CSDC接受度` 的连锁推进

- 娱乐产业的泡沫周期：`娱乐产业景气度`从"逆势增长"到"泡沫破裂"的拐点信号
- 区域分化的细化：`区域分化`和`民众心态`应体现不同地区的不同命运
- 互联网状况的特殊意义：断网=特别大选选举危机

#### Step 2.4 — 角色情况展开

> 阶段一已建立包含8个子维度的角色情况 Schema。阶段二的重点是**完善各字段的填写规范**，确保AI在群像剧中能一致地追踪每个角色的状态变化。

##### 2.4.1 — 政治卷入度的跃迁

`政治卷入度` 五个等级之间的**跃迁**是关键叙事节点：

| 跃迁方向            | 示例场景                                           | 叙事意义             |
| ------------------- | -------------------------------------------------- | -------------------- |
| 未卷入 → 间接影响   | 停电导致无法开工/上课、米价太贵吃不起饭            | 时代碾压日常         |
| 间接影响 → 被动卷入 | 被特别大选推上职位、家人被绑架、Live House被征用   | 被迫进入政治世界     |
| 被动卷入 → 主动参与 | 意识到不能逃避后开始反抗、组织互助网络、加入自警团 | 角色觉醒             |
| 主动参与 → 深度卷入 | 成为阁僚/自警团领袖/势力代理人                     | 成为权力游戏的一部分 |
| 深度卷入 → 被动卷入 | 被罢免/被架空/组织被镇压                           | 权力反噬             |
| 任何等级 → 未卷入   | 退出政治、躲回日常（可能只是暂时的）               | 逃避——但时代不允许   |

##### 2.4.2 — 关键字段填写规范

- **`卷入原因`**：必须清晰追溯"为什么是她"——被互联网投票推上职位 / 被指定为内阁 / 家人被牵连 / 地区成为争夺焦点 / 经营的店铺被征用 / 主动参选
- **`更关心的事`**：必须与政治无关（至少表面上无关），反映角色核心身份——外务大臣更关心下周Live、Live House老板更关心食材到货，这种反差就是2039的戏剧张力
- **`关系动态`**：追踪跨团体、跨学校、跨地域的关系变化，特别关注因政治立场产生的友谊裂痕或意外联盟
- **`所属团体动态`**：她所在的乐队/店铺/家庭/组织是否仍正常运作？是否因时局被迫转型？成员间是否有分歧？
- **`个人剧情线`**：用简短标签概括（如"从逃避到面对""旧时代的幸存者"），帮助AI在多轮对话中保持角色故事一致性

##### 2.4.3 — 角色更新策略

采用"出场即更新"策略：**本轮出场角色**完整更新，**被提及角色**仅更新 `关系动态`/`所属团体动态`，**未出场角色**保留原值。

##### 2.4.4 — 与阁僚 Schema 的数据同步

当角色担任公职时，`各角色情况`（人物视角）与 `日本执政.本届内阁.主要阁僚`（政治视角）存在信息重叠但视角不同。在脚本中监听 `VARIABLE_UPDATE_ENDED` 自动同步安全状况：

```ts
eventOn(Mvu.events.VARIABLE_UPDATE_ENDED, (variables, variables_before) => {
  const data = variables.stat_data;

  // 同步安全状况：阁僚Schema → 各角色情况
  for (const [阁僚名, 阁僚] of Object.entries(data.日本执政?.本届内阁?.主要阁僚 ?? {})) {
    const 角色 = data.各角色情况?.[阁僚名];
    if (角色) {
      if (阁僚.人身安全状况 === '已死亡') 角色.人身安全 = '已死亡';
      else if (阁僚.暗杀威胁等级 === '极高' || 阁僚.暗杀威胁等级 === '高') 角色.人身安全 = '需要安保';
    }
  }
});
```

#### Step 2.5 — 舆论场展开（🆕）

> 阶段一已建立分群体的舆论 Schema（3个跨群体指标 + `z.record()` 按群体索引）。阶段二的重点是**完善各群体的 `核心关切` 和 `动向` 字段的填写规范**，确保AI知道不同群体的舆论如何分化演变。

- 跨群体指标：`社交媒体热点`追踪当前热门话题标签，`互联网直选风向`追踪整体网民情绪基调，`主流媒体立场`追踪各社分化
- 群体划分原则：按地域（都市/农村/港口/冲绳/沼津）+ 按阶层（富裕层/中产/底层）+ 按政治倾向（极右翼/左翼/中间），只追踪当前剧情活跃的群体（通常3-6个）
- 群体间舆论冲突：追踪不同群体对同一事件的对立反应（如「港口居民欢迎CSAT走私粮」vs「农民武装对抗走私」）
- `动向`字段的关键变化信号：从沉默到愤怒、从躺平到组织化、从抗议到武装化、从支持到倒戈
- `政治活跃度`的阈值含义：0-10=沉默躺平，10-30=网络讨论，30-60=街头抗议，60-100=有组织行动/武装化

#### Step 2.6 — 事件队列展开（🆕）

- 近期已发生事件的详细影响追踪
- 潜在危机的触发条件细化
- 事件与政策的因果关系链（事件→政策→效果→新事件）
- 外部势力可能制造的突发事件（暗杀/军事挑衅/经济制裁）

### 阶段三：派生计算与联动（第3轮）

**目标**：利用 `.transform()` 和模块间联动，让Schema自动生成高层次的剧情提示标签。

#### Step 3.1 — 顶级派生字段

在Schema顶层（或各模块内）增加 `$` 前缀的派生字段：

```ts
// 日本执政模块的派生
$执政合法性: z.enum(['合法执政', '争议执政', '合法性危机', '非法执政'])
// 由 首相状态 + 选举进程 + 舆论场.主流媒体立场 + 内阁稳定度 综合推导

$政府控制力: z.enum(['完全控制', '有效治理', '勉强维持', '半瘫痪', '停摆'])
// 由 事务次官会议.实际权力等级 + 各省厅.内阁掌控度（平均值）+ 内阁稳定度 综合推导

// 全局派生
$整体危机等级: z.enum(['正常', '注意', '警戒', '严重', '危急'])
// 由 治安等级 + 通胀等级 + 政治阶段 + 粮食供应 + 事件队列.潜在危机 综合推导

$当前剧情焦点: z.string()
// 根据政治阶段 + 大事件 + 选举进程 + 各角色情况 自动生成一句话提示

$外部压力指数: z.enum(['低', '中', '高', '极高'])
// 由 国际势力动态中各势力的行动能力 + 对日策略 综合推导
```

#### Step 3.2 — 模块间联动（因果链）

核心因果链：**事件 → 决策 → 政策 → 效果 → 反馈 → 新事件**

```
事件队列.潜在危机爆发
     │
     ▼
日本执政.当前政策（新政策出台 / 旧政策受阻）
     │
     ├──→ 日本国内情势（经济民生 + 社会秩序统一变化）
     ├──→ 舆论场（媒体立场/社交媒体热点变化）
     │
     ▼
日本执政.主要政治势力.支持度（执政效果反馈）
日本执政.本届内阁.内阁稳定度（执政效果反馈）
     │
     ▼
国际势力动态（外部势力根据日本政策调整策略）
     │
     ▼
事件队列.潜在危机（外部势力制造新事件）
```

**具体联动规则**：

- `国际势力动态` 变化 → 影响 `日本执政.主要政治势力.支持度`
- `日本国内情势.通胀等级` → 影响 `各角色情况.心理状态` 和 `舆论场`
- `日本国内情势.治安等级` → 影响 `各角色情况.心理状态` 和 `日本国内情势.民众心态`
- `日本国内情势.粮食供应等级` → 影响 `日本国内情势.民众心态` 和 `事件队列.潜在危机`
- `日本国内情势.能源供应等级` → 影响 `日本执政.本届内阁.内阁稳定度`
- `选举进程` 变化 → 自动推进 `世界.政治阶段`
- `选举进程.阶段` 停滞（无人愿出任首相） → 触发 `世界.选举模式` 从「正常首相指名」切换为「2039特别大选」
- `选举进程` 成功产出新首相 → `世界.选举模式` 复位为「正常首相指名」
- `日本执政.当前政策` 变化 → 影响 `日本国内情势` + `舆论场`
- `舆论场`（社会群体情绪分化、反CSAT声浪） → 影响 `日本执政.当前政策.推行状态`
- `事件队列.近期已发生事件` → 影响 `舆论场.社交媒体热点`
- `各角色情况.政治卷入度` 跃迁 → 触发 `各角色情况.关系动态` + `各角色情况.所属团体动态` 连锁更新
- `日本执政.本届内阁.主要阁僚` 中阁僚变动 → 同步更新 `各角色情况` 中对应角色的 `担任公职` + `政治卷入度`
- `日本国内情势.经济状况` 宏观变化 → 影响 `各角色情况.经济状况`（群体性下滑时批量更新）

#### Step 3.3 — 数值范围合理化

统一规范：

- 百分比类（支持度/掌控度/活跃度等）：0~100
- 态度倾向类（对CSAT/对美/意识形态）：-100~100
- 等级类（严重程度）：使用 `z.enum` 分级
- 允许每轮施工中调整数值范围

## 四、注意事项

### Schema 设计范式

1. **优先用 `z.enum` 而非数值范围** —— AI更容易从文本中选择状态，而非精确控制数字。
2. **数值类字段尽量加 `.transform(v => _.clamp(v, min, max))`** —— 防止脏数据。
3. **大量条目用 `z.record()`** —— 追踪日本47个省厅不如追踪当前活跃的几个；并且 `z.record()` 天然支持动态增删条目。
4. **`.describe()` 标注字段语义** —— 帮助AI理解字段含义（如 `z.string().describe('如：俄罗斯GRU')`）。
5. **`.transform()` 做派生，不破坏原始数据** —— 派生字段用 `$` 前缀命名。

### Schema 引用方式

```ts
import { registerMvuSchema } from 'https://testingcf.jsdelivr.net/gh/StageDog/tavern_resource/dist/util/mvu_zod.js';
```

### 类型导出

```ts
export type Schema = z.output<typeof Schema>;
```
