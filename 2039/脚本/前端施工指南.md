# 2039 前端界面 —— 施工指南

## 一、概述

本指南规划 2039 角色卡所需的所有前端界面，包括**状态栏**、**数据仪表盘**、**流式楼层增强**三大类。目标是让玩家在酒馆中**一眼看懂当前政局状态**，同时为"模拟政府执政"的叙事提供沉浸式的视觉反馈。

### 界面全景

```
酒馆界面
├── 聊天区（流式楼层界面）
│   └── [扩展] 消息楼层增强：高亮变量信息、分段折叠、危机预警标签
│
├── 消息楼层前端界面（单个 iframe 主面板）
│   └── [新增] 综合主面板
│       ├── 状态栏：控制台读数 + 批示标签 + 指标矩阵（常驻顶部）
│       └── Tab 切换详情区
│           ├── [Tab] 内阁面板：内阁概况 + 权力对峙 + 阁僚一览
│           ├── [Tab] 省厅面板：各省厅运行状态
│           ├── [Tab] 民生面板：民众生活状况
│           ├── [Tab] 人物面板：关键人物追踪
│           ├── [Tab] 政策追踪器：当前政策生命周期看板
│           ├── [Tab] 国际面板：国际关系动态
│           ├── [Tab] 危机仪表盘：四维雷达 + 潜在危机
│           └── [Tab] 选举看板：候选人排名 + 选举热度
│
├── [后期] 脚本悬浮窗口（脱离消息楼层，全局常驻）
│   └── 衍生内容处理（事件通知、迷你状态球、选举弹幕等）
│
└── 变量数据层（Pinia store）
    └── 2秒轮询同步 stat_data → 主面板自动更新
```

### 核心原则

1. **变量驱动 UI**——所有界面数据来源于 `stat_data`，通过 Pinia store 的2秒轮询自动刷新，不需要手动触发
2. **一眼读懂政局**——状态栏必须紧凑、直观、即时可读，用颜色/图标辅助
3. **渐进式施工**——先做主面板骨架（状态栏 + Tab 容器），再逐个实现各 Tab 详情面板，最后做悬浮窗口
4. **单 iframe 聚合**——所有核心面板整合到同一个 iframe 主面板中，共享一个 Pinia store，避免多 iframe 的数据同步问题
5. **脚本补位**——脱离消息楼层的全局功能（悬浮窗口）用脚本 + jQuery 实现，不依赖 iframe
6. **适配卡片形状**——前端界面在消息楼层中显示，天然适合卡片比例
7. **移动优先**——所有主面板组件先按 320px～430px 宽度设计，再向桌面端扩展；禁止依赖横向并排布局作为唯一显示方式

## 二、技术栈与模式

### 2.1 技术栈

| 技术                    | 用途     | 备注                                |
| ----------------------- | -------- | ----------------------------------- |
| Vue 3 (Composition API) | 组件框架 | 通过 `createApp` 挂载               |
| Pinia                   | 状态管理 | `defineMvuDataStore` 封装了变量轮询 |
| TypeScript              | 类型安全 | 导入 `Schema` 类型                  |
| SCSS (scoped)           | 样式     | 禁止用 `<link>` 引用外部 CSS        |
| Tailwind CSS            | 工具类   | 仅 iframe 宿主可用                  |
| FontAwesome             | 图标     | 免费图标集可用                      |

### 2.2 Pinia Store 模式

每个前端界面必须建立自己的 Pinia store，通过 `defineMvuDataStore` 绑定当前楼层的 MVU 数据：

```ts
// store.ts
import { defineMvuDataStore } from '@util/mvu';
import type { Schema } from '../../2039_mvu/mvu_schema'; // 仅导入类型

export const useDataStore = defineMvuDataStore(
  Schema,  // 传入 Schema（仅用于类型推断和 safeParse）
  { type: 'message', message_id: getCurrentMessageId() }
);
```

**关键特性**：

- `useDataStore().data` 是一个 Vue `Ref<MvuData>`，每2秒自动同步
- Schema 的 `.transform(deriveAll)` 在每次同步时自动执行，派生字段（`$执政合法性` 等）始终最新
- `ignoreUpdates` 机制保证 UI 修改不会触发死循环

### 2.4 iframe 宿主选择

| 宿主类型        | 样式隔离   | Tailwind | 酒馆样式继承 | 适用场景                     |
| --------------- | ---------- | -------- | ------------ | ---------------------------- |
| `iframe` (默认) | ✅ 完全隔离 | ✅ 可用   | ❌ 不继承     | 复杂独立界面（仪表盘、面板） |
| `div`           | ❌ 不隔离   | ❌ 禁止   | ✅ 继承       | 需融入酒馆样式的轻量组件     |

> **推荐**：2039 所有面板界面使用 `iframe` 宿主（样式隔离 + Tailwind 可用）

### 2.5 主面板容器架构（推荐）

为了避免多 iframe 的数据同步问题，**所有核心面板整合到同一个 iframe 主面板中**，通过 Tab 切换展示不同内容。

```
┌──────────────────────────────────────────────────┐
│  主面板（单个 iframe，跟随消息楼层）                │
│  ┌────────────────────────────────────────────┐  │
│  │  状态栏（常驻顶部，紧凑信息条）              │  │
│  │  🔴 看守停摆期 │ 首相：空位 │ ⚠️ 危机：严重 │  │
│  ├───────── Tab 栏 ──────────────────────────┤  │
│  │  [内阁] [省厅] [民生] [人物] [政策] [国际] [危机] [选举]     │  │
│  ├────────────────────────────────────────────┤  │
│  │                                            │  │
│  │  当前 Tab 的内容区域                        │  │
│  │  （点击 Tab 切换，再点同一 Tab 折叠）        │  │
│  │                                            │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

**架构优势**：

| 方案                                 | 优点                                      | 缺点                                                |
| ------------------------------------ | ----------------------------------------- | --------------------------------------------------- |
| ✅ **单 iframe + Tab 切换（本方案）** | 共享一个 store，数据一致；纯 Vue 实现简单 | 单个 iframe 体积略大                                |
| ❌ 多个独立 iframe                    | 职责分离清晰                              | 各自独立 store 不同步；浪费内存和连接；状态同步困难 |

**状态栏与 Tab 区域的交互**：

- 默认：只显示状态栏（最小高度），Tab 区域折叠
- 点击 Tab 标签 → 展开 Tab 区域，显示对应内容
- 点击已激活的 Tab → 折叠 Tab 区域，回到仅状态栏模式
- 这样既"一眼看懂"当前政局，又不会每条消息都占很大空间

## 三、界面详细设计

### 3.0 主面板容器（优先级：🔴 最高，贯穿全流程）

**定位**：所有核心面板的载体。一个 iframe 内包含常驻状态栏 + Tab 切换详情区，所有数据共享同一个 Pinia store。

**App.vue 伪代码结构**：

```vue
<template>
  <div class="main-panel">
    <!-- 状态栏：始终显示在顶部 -->
    <StatusBar />

    <!-- Tab 栏：点击切换详情面板 -->
    <div class="tab-bar">
      <button v-for="tab in tabs" :key="tab.key"
        :class="{ active: currentTab === tab.key }"
        @click="toggleTab(tab.key)">
        {{ tab.label }}
      </button>
    </div>

    <!-- Tab 内容区：仅当展开时显示 -->
    <Transition name="slide">
      <div v-show="expanded" class="tab-content">
        <内阁面板 v-if="currentTab === 'cabinet'" />
        <省厅面板 v-if="currentTab === 'ministry'" />
        <民生面板 v-if="currentTab === 'livelihood'" />
        <人物面板 v-if="currentTab === 'characters'" />
        <政策追踪 v-if="currentTab === 'policy'" />
        <国际面板 v-if="currentTab === 'international'" />
        <危机仪表盘 v-if="currentTab === 'crisis'" />
        <选举看板 v-if="currentTab === 'election'" />
      </div>
    </Transition>
  </div>
</template>
```

**核心状态**：

```ts
const expanded = ref(false);
const currentTab = ref<'cabinet' | 'ministry' | 'livelihood' | 'characters' | 'policy' | 'international' | 'crisis' | 'election'>('cabinet');
const toggleTab = (tab: string) => {
  if (currentTab.value === tab) expanded.value = !expanded.value;
  else { currentTab.value = tab; expanded.value = true; }
};
```

**数据流**：所有 Tab 组件通过 `useDataStore()` 读取同一份 `data`，任何派生字段（`$执政合法性` 等）在所有 Tab 中保持一致。

**移动端适配**：

- Tab 栏使用 `display: flex; overflow-x: auto` 支持横向滚动，按钮最小触摸区域不小于 `44px`
- Tab 内容区在手机端强制单列布局，不使用横向并排
- 详情弹窗在手机端同样使用居中弹窗（宽度自适应 90%~100%，见 3.0 详情弹窗移动端规则）
- 使用 CSS Container Query（`container-type: inline-size`）根据容器宽度切换布局，而非依赖 `window.innerWidth`（因为主面板在 iframe 内）

### 3.2 内阁面板（优先级：🔴 高，第2轮施工）

```vue
    v-if="selectedMinister"
    :职位="selectedMinister"
    :阁僚="阁僚条目.find(([k]) => k === selectedMinister)?.[1]"
    @close="selectedMinister = null"
  />
</template>
```

#### 3.2.5 数据流与 store 处理

```ts
// 内阁面板.vue <script setup>
import { useDataStore } from './store';
const store = useDataStore();
const data = computed(() => store.data as MvuData);

// 阁僚列表：z.record() 需用 _.entries() 过滤
const 阁僚条目 = computed(() => {
  const 阁僚们 = data.value.日本执政?.本届内阁?.主要阁僚 ?? {};
  return _.entries(阁僚们) as [string, typeof 阁僚们[string]][];
});

// 政治势力列表
const 势力条目 = computed(() => {
  const 势力们 = data.value.日本执政?.主要政治势力 ?? {};
  return _.entries(势力们) as [string, typeof 势力们[string]][];
});

// 权力对峙结论
const 对峙结论 = computed(() => {
  const 稳定度 = data.value.日本执政?.本届内阁?.内阁稳定度 ?? 0;
  const 权力等级 = data.value.日本执政?.事务次官会议?.实际权力等级;
  const 首相状态 = data.value.日本执政?.首相状态;
  const 与内阁关系 = data.value.日本执政?.事务次官会议?.与内阁的关系;
  const 当前状态 = data.value.日本执政?.事务次官会议?.当前状态;

  if (首相状态 === '空位') return '无首相状态下，事务次官会议全面接管政务';
  if (稳定度 < 20 && (权力等级 === '主导政府运作' || 权力等级 === '完全替代内阁'))
    return '内阁被严重架空，事务次官会议实际掌权';
  if (稳定度 < 50 && 权力等级 === '主导政府运作')
    return '内阁处于弱势，事务次官会议主导决策';
  if (与内阁关系 === '公开对立' || 与内阁关系 === '操控内阁')
    return '事务次官会议与内阁处于对抗状态，权力争斗激烈';
  if (稳定度 >= 50 && 权力等级 !== '主导政府运作' && 权力等级 !== '完全替代内阁')
    return '内阁与事务次官会议形成均势';
  if (稳定度 >= 70) return '内阁有效掌控政府，事务次官会议处于从属地位';
  if (当前状态 === '决策瘫痪' || 当前状态 === '被架空')
    return '事务次官会议内部混乱，决策能力严重受损';
  return '权力格局研判中';
});

// 弹窗状态
const selectedMinister = ref<string | null>(null);
const openMinisterDetail = (职位: string) => { selectedMinister.value = 职位; };
const closeMinisterDetail = () => { selectedMinister.value = null; };

// 已决议事项（最近3条）
const 最近决议事项 = computed(() => {
  const 事项 = data.value.日本执政?.事务次官会议?.已决议事项 ?? [];
  return _.takeRight(事项, 3);
});
```

#### 3.2.6 空状态与边界情况

所有数据字段都可能为 `undefined`（初始化前/数据不完整时），内阁面板必须优雅处理以下边界场景：

| 场景                          | 判断条件                          | 显示规则                                                           |
| ----------------------------- | --------------------------------- | ------------------------------------------------------------------ |
| **首相空位 + 无阁僚**         | `首相状态 === '空位'` 且阁僚为空  | 内阁概况区显示「内阁尚未组建」占位文案，阁僚一览区显示「暂无阁僚」 |
| **阁僚姓名为空**              | `阁僚.姓名 === ''` 或 `undefined` | 阁僚行姓名栏显示「—」或「空位」，行可点击但弹窗内标记为「未任命」  |
| **政治势力为空**              | `主要政治势力` 为空 record        | 显示「暂无活跃政治势力」，引导文案「政治格局尚未形成」             |
| **事务次官会议数据缺失**      | `事务次官会议` 整个为 `undefined` | 权力对峙区显示「情报不足 — 事务次官会议数据不可用」，三层均折叠    |
| **阁僚已死亡**                | `人身安全状况 === '已死亡'`       | 行变灰 + 删除线 + ☠️ 图标，点击不可展开                             |
| **阁僚已流亡**                | `人身安全状况 === '已流亡'`       | 行变灰 + 斜体 + ✈️ 图标，可展开但弹窗顶部标注「已流亡」             |
| **当前政策为空**              | `当前政策` 为空 record            | 显示「暂无活跃政策 → 去「政策追踪」Tab 查看」                      |
| **所有数据为 null/undefined** | store 初始化前 / stat_data 为空   | 全面板显示「⌛ 等待变量同步…」骨架屏（灰色占位块 + 脉冲动画）       |
| **派生字段异常值**            | `$` 字段为 `undefined`            | 显示「--」作为占位符                                               |
| **稳定度非数字**              | `内阁稳定度` 为 `NaN`             | 进度条显示为 0，数字显示「--」                                     |

### 3.3 省厅面板（优先级：🔴 高，第2轮施工）

**定位**：展示日本政府各省厅的**权力归属**与**运行状态**。核心问题是——「这个省厅到底是谁说了算？内阁阁僚，还是事务次官会议的官僚？」

**与内阁面板的关系**：

- 内阁面板的主角是「内阁这个权力机构」和「阁僚这个人」
- 省厅面板的主角是「官僚机器」——各省厅作为行政实体，其运作状态、派系归属、权力博弈
- 内阁面板的阁僚行显示「对职务理解度」和「履职状态」，省厅面板显示「内阁掌控度 vs 事务次官掌控度」——两者互补，分别从**人的视角**和**机构的视角**看同一件事

**设计原则**：

1. **掌控度对比是核心**——每个省厅的核心信息是「内阁掌控度 vs 事务次官会议掌控度」的对抗条
2. **按关键性分层**——29 个省厅不可能全部平等展示，极高/高级别省厅优先展示，低频/极低省厅默认折叠
3. **与阁僚/政策联动**——省厅行可关联主管阁僚（跳转内阁面板）、关联当前政策（跳转政策追踪）
4. **向下钻取**——点击省厅可展开下属局厅列表

**数据来源模块**：`日本执政.各省厅`（含 运作状态、事务次官会议掌控度、内阁掌控度、对CSAT态度、对美态度、倾向派系、主要下属局厅、当前焦点）

> **⚙️ 配置化设计原则**：由于游戏中可能发生体制改革（省厅合并/拆分/新设、下属机构重组、关键性变更等），本面板的**所有阈值、结论文案、颜色分段、排序规则和匹配逻辑均抽取为可配置常量**，不硬编码在组件中。具体配置项参见 [3.3.9 配置常量与运行时自定义](#339-配置常量与运行时自定义)。

#### 3.3.2 各区域详细设计

##### A. 总览条（始终显示）

一行紧凑的汇总统计，让玩家一眼看懂官僚体系的整体健康状况：

| 统计项               | 数据来源                                           | 显示方式        |
| -------------------- | -------------------------------------------------- | --------------- |
| 省厅总数             | `Object.keys(各省厅).length`（实际有数据的省厅数） | `共 N 省厅`     |
| 正常/半瘫痪/停摆计数 | 遍历 `各省厅` 统计 `运作状态`                      | 彩色圆点 + 数字 |
| 内阁平均掌控度       | `各省厅` 中 `内阁掌控度` 的均值                    | 进度条 + 百分比 |
| 事务次官平均掌控度   | `各省厅` 中 `事务次官会议掌控度` 的均值            | 进度条 + 百分比 |

> **结论判定规则**（所有阈值从 `省厅面板配置` 读取，支持自定义。多条结论同时触发时按严重程度 🔴 > ⚠️ > ✅ 只显示最严重的一条）：
>
> | 条件 | 结论 |
> |------|------|
> | 停摆省厅数 ≥ `省厅面板配置.停摆警戒阈值`（默认 3） | 🔴 多省厅停摆，行政体系部分瘫痪 |
> | 事务次官平均掌控 − 内阁平均掌控 > `省厅面板配置.掌控差警戒阈值`（默认 30） | ⚠️ 官僚体系主导行政 |
> | 内阁平均掌控 > `省厅面板配置.内阁掌控优势阈值`（默认 60） | ✅ 内阁有效掌控官僚体系 |
> | 事务次官平均掌控 > `省厅面板配置.事务次官完全主导阈值`（默认 80） | 🔴 官僚体系完全主导，内阁形同虚设 |
> | 正常省厅数 = 总数 | ✅ 所有省厅正常运作 |

```ts
// 匹配逻辑伪代码（关键词映射从运行时配置读取）
import { use省厅面板配置 } from '../../../src/2039_ui/主面板/use省厅面板配置';

const { config } = use省厅面板配置();

const findMinister = (省厅名: string) => {
  const 阁僚们 = data.日本执政?.本届内阁?.主要阁僚 ?? {};
  // 从运行时配置读取关键词映射，体制改革后新增省厅在游戏内设置面板添加即可
  const keyword = config.value.省厅阁僚关键词映射[省厅名]
    ?? 省厅名.replace(/[省厅委员会]/g, '');  // fallback：去掉常见后缀
  for (const [职位, 阁僚] of Object.entries(阁僚们)) {
    if (职位.includes(keyword)) return { 职位, ...阁僚 };
  }
  return null;
};
```

- 匹配到时：显示「主管阁僚：丰川祥子 ▸」（点击跳转内阁面板并打开该阁僚详情弹窗）

- 阁僚已死亡/流亡时：显示删除线姓名 + 对应图标（☠️ / ✈️）
- **体制改革适配**：新增省厅时只需在 `省厅面板配置.省厅阁僚关键词映射` 中添加条目；未配置时自动使用省厅名去掉「省」「厅」「委员会」后缀作为 fallback 关键词

##### G. 排序选项

省厅列表默认按**掌控度差**（事务次官掌控度 − 内阁掌控度）降序排列，同时支持以下排序方式切换（单选，在筛选栏旁以小型下拉或按钮组展示）：

| 排序维度         | 排序键                               | 说明                                                                                                      |
| ---------------- | ------------------------------------ | --------------------------------------------------------------------------------------------------------- |
| 掌控度差（默认） | `(事务次官掌控度 − 内阁掌控度)` 降序 | 差越大 = 越被架空，越靠前越「危险」                                                                       |
| 关键性           | `省厅关键性权重[省厅名]` 降序        | 极高→高→中→低频→极低（权重从 `省厅面板配置.关键性排序权重` 读取，默认极高=5, 高=4, 中=3, 低频=2, 极低=1） |
| 运作状态         | 停摆→半瘫痪→正常 升序                | 问题省厅优先展示                                                                                          |
| 内阁掌控度       | `内阁掌控度` 升序                    | 最无实权的排前面                                                                                          |
| 事务次官掌控度   | `事务次官会议掌控度` 降序            | 官僚控制最强的排前面                                                                                      |
| 对CSAT态度       | `对CSAT态度` 绝对值降序              | 态度最极端的排前面（无论亲或反）                                                                          |
| 对美态度         | `对美态度` 绝对值降序                | 同上                                                                                                      |
| 派系分组         | `倾向派系` 字母序 → 组内掌控度差降序 | 同一派系的省厅聚合展示                                                                                    |

> **⚙️ 配置化**：默认排序方式通过 `省厅面板配置.默认排序` 配置。上表 8 种排序为固定可选维度，如有增减需求需修改源码。

#### 3.3.5 数据流与 Store 处理

```ts
// 省厅面板.vue <script setup>
import { useDataStore } from './store';
import { 省厅关键性, 省厅与下属机构 } from '../../../src/2039_mvu/constants';
const store = useDataStore();
const data = computed(() => store.data as MvuData);

// 省厅条目：过滤 + 排序
const 省厅条目 = computed(() => {
  const 各省厅 = data.value.日本执政?.各省厅 ?? {};
  let entries = _.entries(各省厅) as [string, typeof 各省厅[string]][];

  // 筛选
  if (filterCriticality.value !== '全部') {
    if (filterCriticality.value === '低频及以下') {
      entries = entries.filter(([name]) =>
        ['低频', '极低'].includes(省厅关键性[name] ?? '极低')
      );
    } else {
      entries = entries.filter(([name]) => 省厅关键性[name] === filterCriticality.value);
    }
  }
  if (filterStatus.value !== '全部') {
    entries = entries.filter(([, d]) => d.运作状态 === filterStatus.value);
  }

  // 排序：掌控度差降序（事务次官掌控度 - 内阁掌控度）
  entries.sort((a, b) => {
    const diffA = (a[1].事务次官会议掌控度 ?? 0) - (a[1].内阁掌控度 ?? 0);
    const diffB = (b[1].事务次官会议掌控度 ?? 0) - (b[1].内阁掌控度 ?? 0);
    return diffB - diffA;
  });

  return entries;
});

// 低频省厅分离（默认折叠）
const 高频省厅 = computed(() => 省厅条目.value.filter(([name]) =>
  ['极高', '高', '中'].includes(省厅关键性[name] ?? '极低')
));
const 低频省厅 = computed(() => 省厅条目.value.filter(([name]) =>
  ['低频', '极低'].includes(省厅关键性[name] ?? '极低')
));

// 总览统计
const 总览统计 = computed(() => {
  const 各省厅 = data.value.日本执政?.各省厅 ?? {};
  const values = Object.values(各省厅);
  return {
    总数: values.length,
    正常: values.filter(v => v.运作状态 === '正常').length,
    半瘫痪: values.filter(v => v.运作状态 === '半瘫痪').length,
    停摆: values.filter(v => v.运作状态 === '停摆').length,
    内阁平均掌控度: _.round(_.mean(values.map(v => v.内阁掌控度 ?? 0))),
    事务次官平均掌控度: _.round(_.mean(values.map(v => v.事务次官会议掌控度 ?? 0))),
  };
});

// 主管阁僚匹配（关键词映射从运行时配置读取）
import { use省厅面板配置 } from '../主面板/use省厅面板配置';

const { config } = use省厅面板配置();

const findMinister = (省厅名: string) => {
  const 阁僚们 = data.value.日本执政?.本届内阁?.主要阁僚 ?? {};
  const keyword = config.value.省厅阁僚关键词映射[省厅名]
    ?? 省厅名.replace(/[省厅委员会]/g, '');
  for (const [职位, 阁僚] of Object.entries(阁僚们)) {
    if (职位.includes(keyword)) return { 职位, ...阁僚 };
  }
  return null;
};

// 活跃政策计数
const activePolicyCount = (省厅名: string) => {
  const 政策们 = data.value.日本执政?.当前政策 ?? {};
  return Object.values(政策们).filter(p => p.主管省厅 === 省厅名).length;
};

// 下属局厅展开状态
const expandedMinistry = ref<string | null>(null);

```

```
                    │  MVU 变量（运行时配置）   │

                    └──────────┬───────────────┘

                    ┌──────────────────────────┐
```

#### 三、组件中读取配置

组件通过自定义 composable `use省厅面板配置` 统一读取配置——优先从 MVU 变量读取，不存在时降级为 TS 默认值：

```ts
// src/2039_ui/主面板/use省厅面板配置.ts
import { computed } from 'vue';
import { useDataStore } from './store';
import { 省厅面板默认配置 } from './省厅面板默认配置';
import type { 省厅面板配置类型 } from '../../2039_mvu/mvu_schema';

export function use省厅面板配置() {
  const store = useDataStore();
  const data = store.data as any;

  /** 实际生效的配置：MVU 变量 → TS 默认值 fallback */
  const config = computed<省厅面板配置类型>(() => ({
    ...省厅面板默认配置,
    ...(data.value?.世界?.省厅面板配置 ?? {}),
  }));

  /** 仅读取配置（不变），不修改 */
  return { config };
}
```

**组件中使用**：

```ts
// 省厅面板.vue 中使用
import { use省厅面板配置 } from './use省厅面板配置';

const { config } = use省厅面板配置();

// 总览条结论判定（从运行时配置读取）
const 总览结论 = computed(() => {
  const stats = 总览统计.value;
  if (stats.停摆 >= config.value.停摆警戒阈值)
    return '🔴 多省厅停摆，行政体系部分瘫痪';
  if ((stats.事务次官平均掌控度 - stats.内阁平均掌控度) > config.value.掌控差警戒阈值)
    return '⚠️ 官僚体系主导行政';
  // ...
});

// 派系颜色（从运行时配置读取，fallback 灰色）
const getFactionColor = (派系名: string): string =>
  config.value.派系颜色映射[派系名] ?? '#64748b';

// 态度色块颜色
const getAttitudeColor = (value: number): string => {
  const seg = config.value.态度色段.find(s => value >= s.min && value <= s.max);
  return seg?.color ?? '#64748b';
};
```

**数据流示意**：

```
MVU 变量（世界.省厅面板配置）
         │ 每 2 秒同步
         ▼
Pinia store.data
         │ Vue computed 展开
         ▼
config.value = { ...省厅面板默认配置, ...store.data.世界?.省厅面板配置 }
         │ 响应式绑定
         ▼
所有组件自动刷新
```

> MVU 变量中缺失的字段会自动用 TS 默认值补全（通过 `...` 展开顺序），因此第一次使用时即使 `世界.省厅面板配置` 为空对象，所有字段也有合理值。

#### 五、写入 MVU 变量

设置面板点击「保存配置」后，通过 Mvu API 将配置写回 MVU 变量：

```ts
// src/2039_ui/主面板/省厅面板设置.vue（片段）

import { ref, computed } from 'vue';
import { useDataStore } from './store';
import { use省厅面板配置 } from './use省厅面板配置';

const { config } = use省厅面板配置();

/** 本地编辑的配置副本（提交前暂存） */
const draftConfig = ref({ ...config.value });

/** 保存配置到 MVU 变量 */
const saveConfig = async () => {
  // 1. 将配置写入 MVU 变量
  await _.set('世界.省厅面板配置', draftConfig.value);

  // 2. 触发 AI 感知：在消息中追加一条系统提示
  await _.set('世界.省厅面板配置.上次修改时间', new Date().toISOString());
  await _.set('世界.省厅面板配置.上次修改人', '玩家');

  // 3. 关闭弹窗（2 秒后 Pinia store 自动同步）
  emit('close');
};
```

#### 七、体制改革场景下的运行时修改

体制改革发生时，玩家无需改代码、无需重编译，直接在游戏内通过 ⚙️ 设置面板修改：

| 改革场景                       | 设置面板操作                                                        |
| ------------------------------ | ------------------------------------------------------------------- |
| 新增「数字厅」                 | 打开设置 → 省厅阁僚关键词映射 → [+ 新增映射] → 输入 `数字厅 → 数字` |
| 撤销「复兴厅」                 | 无需任何操作——数据中不再出现则自动排除                              |
| 外务省与经产省合并             | 省厅阁僚关键词映射：删除旧映射，新增 `对外经济省 → 对外经济`        |
| 内阁重获主导权，降低预警敏感度 | 设置中调高 `架空警戒阈值`、`官僚锁死阈值` → 保存                    |
| 新增「对欧盟态度」外交维度     | 见下方「新增维度的扩展流程」                                        |
| 某派系瓦解，去除预警           | 设置 → 启用预警 → 取消勾选 `派系孤立` → 保存                        |
| 调整排序默认方式               | 设置 → 默认排序方式 → 下拉选 `关键性` → 保存                        |
| 手机端改回完整信息             | 设置 → 手机端默认全部折叠 → `否` → 保存                             |
| 派系颜色不好看                 | 设置 → 派系颜色 → 点击色块 → 取色器选新色 → 保存                    |
| 调整态度色段分界               | 设置 → 态度色段 → 编辑区间值或颜色 → 保存                           |

**新增「对欧盟态度」维度的扩展流程**：

```
1. 在 Schema 的各省厅数据中新增「对欧盟态度」字段
   （这是 Schema 变更，需要一次编译）
2. 在 省厅面板设置弹窗 → 态度色段 → 确认已有色段覆盖新字段
   （色段是通用的 -100~+100 映射，无需额外配置）
3. 在 省厅面板设置弹窗 → 详情弹窗区域排序
   → 在『外交取向』区域中，新增的「对欧盟态度」自动显示
   （如果外交部取向区域支持动态渲染所有态度字段）
4. 可选：在 省厅面板设置弹窗 → 详情弹窗区域排序
   → 将新增的「对欧盟态度」添加到显示顺序中
```

> **核心原则**：组件只负责「如何渲染」，配置负责「渲染什么、何时渲染、以什么阈值渲染」。任何可能因游戏进程而变化的数值、文案、映射关系，一律从运行时配置读取，通过前端设置面板随时修改。

### 3.4 民生面板（优先级：🟡 中，第3轮施工）

**定位**：以"老百姓视角"展示日本民众的生活状况——物价、就业、治安感知、社会情绪等民生指标。与内阁面板（权力视角）和省厅面板（机构视角）互补，民生面板回答一个问题：**"普通人现在过得怎么样？"**

**与内阁面板 / 省厅面板的关系**：

| 面板     | 视角         | 核心问题                       |
| -------- | ------------ | ------------------------------ |
| 内阁面板 | 权力机构视角 | 内阁和事务次官会议谁说了算？   |
| 省厅面板 | 官僚机器视角 | 每个省厅被谁掌控？             |
| 民生面板 | 老百姓视角   | 物价涨了吗？有饭吃吗？安全吗？ |

**设计原则**：

1. **以人为本**——不追求政治分析的"冷酷精准"，而是让玩家直观感受社会温度
2. **分类折叠**——~36 个字段不可能全部平铺，按 7 个逻辑分组折叠，每组显示一行摘要
3. **总览条常驻**——最关键的 5 项指标（物价、治安、粮食、能源、民心）始终可见
4. **趋势可视化**——许多字段是变化趋势（恶化 / 改善），用方向箭头 ▲▼ 表达
5. **颜色即情绪**——暖色=痛苦加剧，冷色=状况良好，灰色=中性/不变

**数据来源模块**：`日本国内情势`（含 金融货币、物价民生、粮食能源、利益集团、社会安全网、教育娱乐、治安执法、人口住房、基础设施医疗、区域分化与社会心理，共 10 个子分组，约 36 个字段）

**交互行为**：

- 默认：仅展开「经济民生」分组，其余折叠（可在后续根据玩家偏好记忆展开状态）

- 点击已展开的分组 → 折叠（此时所有分组折叠，仅剩总览条可见）
- 每个折叠分组的摘要行显示该组最关键 2-3 个字段的状态，帮助玩家快速扫描

#### 3.4.5 数据流与 Store 处理

```ts
// 民生面板.vue <script setup>
import { useDataStore } from './store';
const store = useDataStore();
const data = computed(() => store.data as MvuData);

// 民生数据（带类型安全访问）
const livelihood = computed(() => data.value.日本国内情势 ?? {});

// 手风琴状态：当前展开的分组 key（null = 全部折叠）
const expandedSection = ref<string | null>('经济民生'); // 默认展开经济民生

const toggleSection = (key: string) => {
  expandedSection.value = expandedSection.value === key ? null : key;
};

// 分组定义
const sections = [
  { key: '经济民生', icon: '💴', title: '经济民生' },
  { key: '社会秩序', icon: '🛡️', title: '社会秩序' },
  { key: '利益集团', icon: '🏛️', title: '利益集团' },
  { key: '社会安全网', icon: '🍱', title: '社会安全网' },
  { key: '教育与娱乐', icon: '🎵', title: '教育与娱乐' },
  { key: '区域分化', icon: '🗺️', title: '区域分化' },
  { key: '社会心理', icon: '💭', title: '社会心理' },
];

// 折叠摘要行文本生成（每个分组取最严重的 2-3 个状态）
const sectionSummary = computed(() => {
  const d = livelihood.value;
  return {
    '社会秩序': summarizeSection(d, [
      { field: '治安等级', label: '治安' },
      { field: '夜间治安', label: '夜间' },
      { field: '医疗供应等级', label: '医疗' },
    ]),
    '利益集团': summarizeSection(d, [
      { field: '农协状态', label: '农协' },
      { field: '极道影响力', label: '极道', isNumber: true },
      { field: '自警团活跃度', label: '自警团', isNumber: true },
    ]),
    '社会安全网': summarizeSection(d, [
      { field: '生命维持餐运作状态', label: '生命维持餐' },
      { field: '生命维持餐依赖趋势', label: '依赖' },
    ]),
    '教育与娱乐': summarizeSection(d, [
      { field: '教育状况', label: '教育' },
      { field: '娱乐产业景气度', label: '娱乐' },
      { field: 'LiveHouse运营状态', label: 'LiveHouse' },
    ]),
    '区域分化': `${(d.区域分化 ?? []).map(r => r.区域).join('·')} 共${(d.区域分化 ?? []).length}区域`,
    '社会心理': `民众心态：${(d.民众心态 ?? '--').slice(0, 12)}…`,
  };
});

// 摘要生成辅助函数：取前 3 个非正常项
function summarizeSection(
  d: Record<string, any>,
  fields: { field: string; label: string; isNumber?: boolean }[]
): string {
  const items = fields
    .map(f => ({ label: f.label, value: d[f.field] }))
    .filter(item => item.value !== undefined && item.value !== '正常' && item.value !== '无' && item.value !== '极少');
  return items.slice(0, 3).map(i => `${i.label}${i.value}`).join(' · ') || '正常';
}
```

#### 3.4.6 空状态与边界情况

| 场景                           | 判断条件                          | 显示规则                                                        |

##### 搜索功能

在总览条下方增加关键词搜索框，支持模糊匹配：

```
🔍 [搜索民生指标...]  [✕ 清除]
```

| 特性       | 说明                                                                                             |
| ---------- | ------------------------------------------------------------------------------------------------ |
| 搜索范围   | 字段名（如「通胀」「治安」）、枚举值（如「严重通胀」「恶化」）、文本内容（极道动态、民众心态等） |
| 匹配方式   | 模糊包含匹配（不区分大小写），输入即搜（debounce 200ms）                                         |
| 搜索结果   | 高亮匹配到的关键词，自动展开所在分组；无匹配时显示「未找到匹配的指标」空状态占位                 |
| 清除       | 搜索框右侧 ✕ 按钮清除搜索词；搜索词为空时 ✕ 隐藏                                                 |

#### 3.4.10 前端衍生计算字段

**定位**：民生面板中的原始 MVU 数据（如 `大米价格倍数: 5`）对玩家来说不够直观——"5 倍"到底意味着什么？本节定义一组**纯前端计算**的衍生字段，将抽象数值转化为玩家能直观感知的**具体生活场景数据**。所有衍生字段通过 Vue `computed()` 计算，**不修改 MVU 变量、不涉及 Schema 变更**。

**设计原则**：

1. **不写回 MVU**——衍生字段只读，仅用于前端展示
2. **具体化抽象数值**——系数变价格、占比变人数、等级变描述
3. **常量集中管理**——基准值（如 2036 年米价）定义为常量，方便统一调整
4. **容错**——源数据缺失时衍生字段优雅降级为 `--`

##### B. 各衍生字段详细设计

###### B1. 大米单价（日元/kg）

**目的**：让"米价×5"变成玩家能感知的"一公斤大米卖 2500 日元"。

**常量定义**：

```ts
// src/2039_ui/主面板/民生常量.ts
/** 2036 年（战前/基准年）大米零售均价：日元/kg */
export const 基准米价_日元每公斤 = 500;
```

**计算公式**：

```ts
const 大米单价 = computed(() => {
  const 倍数 = data.value.日本国内情势?.大米价格倍数;
  if (倍数 == null || isNaN(倍数)) return null;
  return Math.round(倍数 * 基准米价_日元每公斤);
});
```

**显示方式**：在物价与就业卡片中，大米价格倍数行的下方以灰色小字追加一行：

```
米价 ×5  ≈ ¥2,500/kg
```

| 倍数 | 单价         | 颜色 | 感受         |
| ---- | ------------ | ---- | ------------ |
| 1-2  | ¥500-1,000   | 绿   | 正常         |
| 3-4  | ¥1,500-2,000 | 黄   | 有点贵       |
| 5-7  | ¥2,500-3,500 | 橙   | 吃不起       |
| ≥8   | ¥4,000+      | 红   | 米已成奢侈品 |

###### B2. 日元实际购买力

**目的**：用一句话让玩家感知通胀的严重程度。

**计算公式**：

```ts
const 购买力百分比 = computed(() => {

```

###### B3. 单人月均主食开销

**目的**：将"米价×5"拉近到"一个人一个月光吃米饭就要花多少钱"的个人视角。

**常量定义**：

```ts
/** 成人日均大米消费量（kg），按日本传统饮食约150g/日估算 */
export const 日均大米消费_kg = 0.15;
```

**计算公式**：

```ts
const 月均主食开销 = computed(() => {
  const 单价 = 大米单价.value;
  if (单价 == null) return null;
  return Math.round(单价 * 日均大米消费_kg * 30);
});
```

**显示方式**：在大米单价行下方以更小的灰色字显示：

```
≈ 单人月均主食 ¥11,250
```

> 此数字仅为主食（米饭）开销，不含配菜、调味料、水电燃气等。实际饮食开销通常为此数的 2-3 倍。

###### B4. 粮食正规渠道占比

**目的**：将"走私粮食占比 45%"转化为"正规渠道粮食仅剩 55%"，更直观地体现粮食体系的崩溃程度。

```ts
const 正规渠道占比 = computed(() => {
```

###### B5. 警察净效能

**目的**：警察士气和腐败程度是两个独立数值，玩家难以综合判断。净效能将其合并为一个直观数字。

**计算公式**：

```ts
const 警察净效能 = computed(() => {
  const 士气 = data.value.日本国内情势?.警察士气;
  const 腐败 = data.value.日本国内情势?.警察腐败程度;

});
```

###### B6. 社会痛苦指数（Misery Index）

**目的**：将失业率 + 通胀两个维度的痛苦量化成一个 0-125 的分数，给玩家一个**总览条级别的快照**。

**通胀痛苦分映射**：

```ts
const 通胀痛苦分映射: Record<string, number> = {
  '稳定': 0,
  '温和通胀': 3,
  '严重通胀': 7,
  '恶性通胀': 15,
  '货币崩溃': 25,
};
```

**计算公式**：

```ts
const 社会痛苦指数 = computed(() => {
  const 失业 = data.value.日本国内情势?.失业率;
  const 通胀 = data.value.日本国内情势?.通胀等级 as string;
  if (失业 == null || isNaN(失业)) return null;
  const 通胀分 = 通胀痛苦分映射[通胀] ?? 0;
  return Math.round(失业 + 通胀分);
});
```

**显示方式**：在民生总览条的最右侧（或民心标签的 Tooltip 中）显示：


| 21-35  | `#eab308` 黄 | 社会承压中   |
| 36-50  | `#f97316` 橙 | 普遍憤怒     |
| 51-125 | `#ef4444` 红 | 社会濒临崩溃 |

###### B7. 生命维持餐覆盖估算

**目的**：将"依赖趋势：扩大中"这种定性描述转化为一个预估的覆盖比例，让玩家感知生命维持餐体系在社会中的实际渗透程度。

**计算公式**（查表法）：

```ts
const 生命维持餐覆盖率 = computed(() => {
  const 状态 = data.value.日本国内情势?.生命维持餐运作状态 as string;
  const 趋势 = data.value.日本国内情势?.生命维持餐依赖趋势 as string;

  // 查表：基于运作状态 + 依赖趋势估算覆盖率
  if (状态 === '已崩溃') return '已停止';
  if (状态 === '全面停供') return '仅剩极少数站点';

  if (趋势 === '缩减中') return '约 15-25% 城市人口';
  if (趋势 === '稳定') {
    if (状态 === '正常运作') return '约 25-35% 城市人口';
    if (状态 === '物资紧缺') return '约 30-40% 城市人口';
    return '约 35-45% 城市人口';
  }
  if (趋势 === '扩大中') {
    if (状态 === '正常运作') return '约 40-55% 城市人口';
    if (状态 === '物资紧缺') return '约 50-65% 城市人口';
    return '约 60-75% 城市人口';
  }
  if (趋势 === '濒临崩溃') return '超过 70% 城市人口（配给濒临中断）';

  return '--';
});
```

**显示方式**：在社会安全网卡片中，生命维持餐依赖趋势下方以灰色小字显示：

```
依赖扩大中 ▲  覆盖约 40-55% 城市人口
```

##### C. 衍生字段的数据流

```
MVU 变量（原始数据）
  │ 如：大米价格倍数 = 5
  │     走私粮食占比 = 45
  │     警察士气 = 30, 警察腐败程度 = 55
  │
  │ 每 2 秒同步
  ▼
Pinia store.data
  │
  │ Vue computed()
  ▼
衍生字段
  │ 大米单价 = 5 × 500 = ¥2,500/kg
  │ 购买力 = floor(100/5) = 20%
  │ 正规渠道 = 100 - 45 = 55%
  │ 净效能 = 30 - 55 = -25
  │ 痛苦指数 = 15 + 7 = 22
  │
  │ 响应式绑定
  ▼
民生面板 UI 自动刷新
  │ 零额外代码 —— computed 自动追踪依赖
```

**关键特性**：

- 衍生字段是**纯响应式计算**——源数据变化时 UI 自动更新，无需手动触发
- 所有计算在 `computed()` 中完成，不会产生额外的 MVU 写入
- 常量（基准米价、日均消费等）在独立的 `民生常量.ts` 文件中集中管理，方便调整
- 源数据缺失时所有衍生字段优雅降级为 `null` → UI 显示 `--`

#### 3.4.12 异常预警标记

当民生面板中的多个数据同时满足特定极端组合条件时，在总览条或对应分组上叠加视觉预警。

| 预警类型       | 触发条件                                                | 视觉表现                                                   |
| -------------- | ------------------------------------------------------- | ---------------------------------------------------------- |
| 🔴 社会崩溃     | 粮食=饥荒边缘 且 能源=崩溃                              | 总览条 2px 红色脉冲边框 + 总览条右侧「⚠️ 社会崩溃」标签     |
| 🔴 经济死亡螺旋 | 通胀=货币崩溃 且 失业率>35%                             | 经济民生卡片红色脉冲边框 + 分组标题旁「⚠️ 经济死亡」标签    |
| 🟠 执法系统崩溃 | 治安=无政府状态 或（警察净效能<0 且 夜间治安=宵禁状态） | 社会秩序分组标题旁 ⚠️ 图标 + 治安卡片淡红背景               |
| 🟠 粮食体系失控 | 走私粮占比>60% 且 农协=濒临解体/已解体                  | 粮食卡片叠加淡红背景 `rgba(239,68,68,0.04)` + 边框红色脉冲 |
| 🟡 末日狂欢     | 娱乐=爆发式增长/泡沫化 且 粮食≤危机                     | 教育与娱乐分组标题旁 ⚠️ 图标 + 娱乐卡片黄色脉冲边框         |
| 🟡 人口塌方     | 富裕阶层=已基本出逃 且 无家可归者=随处可见/已成社会常态 | 人口住房子卡片叠加淡黄背景 + 分组摘要行黄色文字            |

**预警叠加规则**：

- 同一区域同时满足多个预警时，按严重程度 **🔴 > 🟠 > 🟡** 只显示最严重的一个
- 预警标签停留在对应区域直到条件解除；条件解除后标签淡出（0.5s）
- 预警触发时对应的分组标题行短暂闪烁 1s，提醒玩家注意
- 所有预警的触发条件阈值设计为常量，集中在 `民生常量.ts` 中管理

##### 组件中读取配置

```ts
// src/2039_ui/主面板/use民生面板配置.ts
import { computed } from 'vue';
import { useDataStore } from './store';
import { 民生面板默认配置 } from './民生面板默认配置';
import type { 民生面板配置类型 } from '../../2039_mvu/mvu_schema';

export function use民生面板配置() {
  const store = useDataStore();
  const data = store.data as any;

  const config = computed<民生面板配置类型>(() => ({
    ...民生面板默认配置,
    ...(data.value?.世界?.民生面板配置 ?? {}),
  }));

  return { config };
}
```

##### 应用示例

```ts
// 民生面板.vue 中使用配置
import { use民生面板配置 } from './use民生面板配置';

const { config } = use民生面板配置();

// 衍生计算使用可配置的基准值
const 大米单价 = computed(() => {
  const 倍数 = data.value.日本国内情势?.大米价格倍数;
  if (倍数 == null) return null;
  return Math.round(倍数 * config.value.基准米价_日元每公斤);
});

// 预警判定使用可配置的阈值
const 社会崩溃预警 = computed(() => {
  if (!config.value.启用预警.社会崩溃) return false;
  const 粮食 = data.value.日本国内情势?.粮食供应等级;
  const 能源 = data.value.日本国内情势?.能源供应等级;
  return 粮食 === config.value.预警阈值.社会崩溃_粮食等级
    && 能源 === config.value.预警阈值.社会崩溃_能源等级;
});
```

- 民生面板维护各自的选中状态 ref：`selectedRegion`, `selectedInterestGroup`, `selectedPsychology`
- 弹窗组件通过 `v-if` 条件渲染

- 点击 ✕ / 遮罩 / ESC 关闭
- 弹窗与手风琴展开/折叠互斥——打开弹窗时自动冻结手风琴交互

```vue
    @click="selectedRegion = 区域.区域"
  />

  <!-- 弹窗 -->
```

### 3.11 悬浮窗口（优先级：🟢 低，第6轮施工）

**定位**：脱离消息楼层的全局悬浮窗口，用于处理衍生内容——阁僚详情弹窗、政策详情抽屉、事件通知等主面板中放不下的次要信息。

**实现方式**：**脚本**项目（仅有 `index.ts`，没有 `index.html`），用 jQuery 在酒馆 DOM 中注入 `position: fixed` 定位元素，通过 `Mvu.getMvuData()` 或 `getVariables()` 读取数据。

**为什么不用 iframe**：

| 方式                          | 说明                        | 问题                                               |
| ----------------------------- | --------------------------- | -------------------------------------------------- |
| ❌ iframe 内 `position: fixed` | iframe 嵌入在消息楼层容器中 | `fixed` 受限于 iframe 边界，无法溢出到整个酒馆页面 |
| ❌ 多 iframe 间 `postMessage`  | 主面板与悬浮 iframe 通信    | 架构复杂，延迟高                                   |
| ✅ **脚本 + jQuery**           | 脚本直接操作酒馆 DOM        | 简单直接，数据同源                                 |

**适用场景**：

- **政策详情抽屉**——展示政策的完整副作用链条和对各势力的影响（政策详情在内阁面板的摘要区不完整展示）
- **事件通知横幅**——危机升级时在页面顶部弹出警示横幅
- **迷你状态悬浮球**——酒馆页面角落常驻简化的状态指示器
- **选举实时弹幕**——选举期间的舆论动态滚动

**伪代码示例**：

```ts
// src/2039_悬浮窗口/index.ts
import FloatingWindow from './FloatingWindow.vue';

$(() => {
  // 在酒馆页面中注入一个 fixed 定位的容器
  const container = $('<div id="th2039-floating-window"></div>').appendTo('body');

  // 挂载 Vue 应用（mount 到 body 内的 DOM 元素，而非 iframe）
  const app = createApp(FloatingWindow);
  app.mount(container[0]);

  $(window).on('pagehide', () => {
    app.unmount();
    container.remove();
  });
});
```

**关键技术点**：

- 脚本注入的元素使用 `position: fixed; z-index: 9999;` 脱离正常文档流
- 数据通过 `Mvu.getMvuData()` 直接读取（不经过 `defineMvuDataStore`）
- 如需响应式更新，可在脚本内建立简单的 `setInterval` 轮询，或监听 MVU 事件 `Mvu.events.VARIABLE_UPDATE_ENDED`
- 脚本内也可以使用 Vue + Pinia，但需手动挂载到 `body` 而非 iframe 内
- 注意酒馆主题切换时悬浮窗口的颜色适配

#### 移动端适配

手机端悬浮窗口容易遮挡输入框、按钮和消息内容，需要特殊处理：

| 桌面端         | 手机端                             |
| -------------- | ---------------------------------- |
| 右下角悬浮窗口 | 右下角悬浮球（默认收起）           |
| 可展开侧边面板 | 点击后底部抽屉                     |
| 可显示多条通知 | 通知自动合并，只显示最高危机       |
| 可拖动         | 不建议默认拖动，避免和页面滚动冲突 |

手机端默认只显示一个小悬浮球：

```
● 2039
```

点击后展开为底部抽屉，而不是右侧面板。

**关键规则**：

- 悬浮球位置：右下角偏上，避免遮挡酒馆输入框
- `bottom` 不要固定死，应考虑输入区高度
- 展开时覆盖半屏，而不是全屏常驻
- 底部抽屉留出安全区：`padding-bottom: env(safe-area-inset-bottom)`
- 关闭按钮明显，点击遮罩也可关闭

## 四、样式指南

### 4.1 配色体系

整体采用**政府指挥中心 / 战略仪表盘**暗色主题：深海军蓝底色（`#0b1121` → `#111827`），叠加经纬网格线营造"战情室"氛围。标签采用左侧竖条批示风格（而非圆角胶囊），标题徽章用红底方章像政府公章。

| 用途          | 颜色      | Tailwind                    |
| ------------- | --------- | --------------------------- |
| 危险/极高     | `#ef4444` | `text-red-500` `bg-red-500` |
| 警告/高       | `#f97316` | `text-orange-500`           |
| 注意/中       | `#eab308` | `text-yellow-500`           |
| 正常/低       | `#22c55e` | `text-green-500`            |
| 信息/进行中   | `#3b82f6` | `text-blue-500`             |
| 特殊/关键节点 | `#8b5cf6` | `text-purple-500`           |
| 稳定          | `#06b6d4` | `text-cyan-500`             |

### 4.2 组件尺寸约束

- 状态栏：最大宽度 100%，高度自适应（约 60-80px）
- 卡片（阁僚/政策）：约 200-280px 宽，高度自适应
- 仪表盘面板：约 350-400px 宽
- 所有面板：禁止使用 `vh` 单位，使用 `aspect-ratio` 控制高度
- 不要有横向滚动条
- 手机端卡片强制 `width: 100%`，不使用固定宽度

### 4.3 动画

- 数值变化：CSS `transition: all 0.3s ease`
- 危机等级升级：进度条红色脉冲动画（`@keyframes pulse-red`）
- 新事件发生：卡片短暂闪烁（黄色边框 1s 后消失）
- 阶段切换：标签 0.3s 渐变切换

### 4.4 手机端适配

- 以 320px 宽度作为最低适配目标，所有核心信息必须在 360px 宽度下无横向滚动
- 状态栏所有 12 项数据指标全部可见：日期行 + 第一行 3 个核心标签（flex-wrap），第二行 2×3 等宽网格
- 状态栏标签使用左侧竖条批示风格（`border-left: 3px`），在手机端缩减为 `2px`
- Tab 栏允许横向滚动，按钮触摸区域不小于 `44px`
- 卡片在手机端强制单列，不使用多列网格
- 详情弹窗在手机端同样使用居中弹窗（宽度自适应 95%），底部抽屉仅作为极窄屏（≤ 320px）备选
- 悬浮窗口在手机端默认收起为悬浮球，点击展开为底部抽屉
- 禁止依赖 `hover` 作为关键交互；所有交互必须支持点击/触摸
- 避免固定宽度，优先使用 `width: 100%`、`max-width`、`clamp()`、`minmax()`
- 不使用 `vh` 作为主高度单位，避免被移动端浏览器地址栏影响
- 使用 CSS Container Query（`container-type: inline-size`）根据容器宽度切换布局，而非依赖 `window.innerWidth`（因为主面板在 iframe 内）
- 激活态必须靠颜色 + 边框 + 字重区分，不只靠颜色（照顾色觉障碍用户）

## 六、关键实现细节

### 6.1 `z.record()` 的 v-for 遍历

由于 `z.record()` 解析后的对象 **不保证** 是普通对象（可能包含原型链属性），遍历时必须过滤：

```vue
<template>
  <div v-for="[职位, 阁僚] in 阁僚条目" :key="职位">
    <!-- 阁僚卡片 -->
  </div>
</template>

<script setup lang="ts">
import { useDataStore } from './store';
const { data } = storeToRefs(useDataStore());

const 阁僚条目 = computed(() => {
  const 阁僚们 = data.value.日本执政?.本届内阁?.主要阁僚 ?? {};
  // _.entries() 只返回 own enumerable properties
  return _.entries(阁僚们) as [string, typeof 阁僚们[string]][];
});
</script>
```

### 6.2 派生字段的访问

派生字段（`$执政合法性` 等）在 `deriveAll` 中通过 `...data` 展开后附加。在 Vue 模板中直接访问：

```vue
<template>
  <span>{{ data.$整体危机等级 }}</span>
</template>
```

但 TypeScript 类型可能不包含 `$` 字段（因为 `Schema.transform()` 的输出类型与输入不同）。解决方案：在 store.ts 中使用时做类型断言，或定义扩展类型。

### 6.3 性能注意事项

- `useDataStore().data` 是深度响应式（`deep: true`），每2秒全量替换
- 不要在 `computed` 中进行重型计算，尽量直接读取
- 大列表（如所有角色情况）考虑虚拟滚动（但2039角色数通常<20，不需要）
- SVG 雷达图避免在 `computed` 中重新生成 path，使用 `watch` + 缓存

### 6.4 空值处理

所有字段都可能为 `undefined`（初始化前/数据不完整时），模板中必须做好兜底：

```vue
<template>
  <span>{{ data.日本执政?.首相姓名 || '空位' }}</span>
  <ProgressBar :value="data.日本执政?.本届内阁?.内阁稳定度 ?? 0" />
</template>
```

### 6.5 容器查询与移动端布局切换

由于主面板在 iframe 内，`window.innerWidth` 反映的是 iframe 自身的宽度而非酒馆页面宽度。因此移动端布局切换应使用 **CSS Container Query** 而非 Media Query：

```scss
// App.vue
.main-panel {
  container-type: inline-size;
  container-name: main-panel;
}

// 状态栏.vue
@container main-panel (max-width: 430px) {
  .status-bar {
    font-size: 12px;
  }
  .status-row-2 {
    display: none; // 手机端默认折叠第二行
  }
  .status-row-2.expanded {
    display: flex;
  }
}

// 内阁面板.vue
@container main-panel (max-width: 430px) {
  .cabinet-grid {
    grid-template-columns: 1fr; // 手机端单列
  }
}

// 危机仪表盘.vue
@container main-panel (max-width: 430px) {
  .radar-chart {
    display: none; // 手机端隐藏雷达图
  }
  .crisis-cards {
    display: block; // 显示替代的纵向指标卡片
  }
}
```

**断点定义**：

| 断点名称 | 容器宽度      | 布局策略             |
| -------- | ------------- | -------------------- |
| `narrow` | ≤ 430px       | 单列、折叠、居中弹窗 |
| `medium` | 431px ~ 768px | 单列但信息更完整     |
| `wide`   | ≥ 769px       | 多列网格、居中弹窗   |

**详情弹窗的移动端适配**：

| 设备               | 详情组件形态        |
| ------------------ | ------------------- |
| 桌面端（`wide`）   | 居中弹窗 / 右侧抽屉 |
| 平板端（`medium`） | 右侧抽屉            |
| 手机端（`narrow`） | 居中弹窗，宽度 95%  |
| 极窄屏（≤ 320px）  | 全屏详情页          |
