# 《群友群穿》MVU 脚本施工指南

> 本指南基于 `世界书/Cot说明.yaml`、`世界书/世界观_总设定.txt`、`世界书/世界观_群友关系网.txt`、`世界书/世界观_群聊分布.txt`、`世界书/USER设定.txt`、`世界书/普通群友_通用模板.txt`、94 份角色档案，以及 `src/2039_mvu/` 参考实现综合编制。
> 目标：为「群友群穿」角色卡搭建一套 MVU 变量框架 + 兜底脚本 + 状态栏界面，支撑 94 人群像在 1905 年中国的群穿叙事。

## 一、项目定位与总体架构

### 1.1 叙事定位

- **题材**：94 名现代群友携基地车群穿至 1905 年（清末）安徽马鞍山附近，就地扎根建立政权。
- **视角**：USER 为「摄像机/观察者」，客观记录群像，不扮演具体角色、不介入剧情。
- **核心张力**：现代人知识红利 vs 1905 落后现实；基地车不可复制 vs 列强与清廷围堵；群友私交 vs 政治派系；理想主义 vs 权术现实。
- **时间锚**：1905.5.10 起步，按现实流速推进，外部大事件（日俄战争结束、废除科举、同盟会成立等）按历史节奏发生。

### 1.2 MVU 在本项目中的职责

| 职责       | 说明                                                                                              |
| ---------- | ------------------------------------------------------------------------------------------------- |
| 状态记忆   | 跨楼层持久化「世界/基地车/科技树/组织/军事/情报/外部/角色/事件」十大模块                          |
| 群像追踪   | 94 名角色的职务、立场、心理、关系、生死、卷入度                                                   |
| 因果兜底   | 脚本对 AI 遗漏的联动更新做自动补全（职务↔角色、死亡↔空缺、危机↔事件、战损↔装备）                  |
| 派生提示   | 通过 `.transform()` 自动生成 `$执政合法性`、`$整体危机等级`、`$军事实力指数` 等高层标签供 AI 参考 |
| 状态栏展示 | 前端界面可视化当前世界状态与角色群像                                                              |

## 二、变量结构设计（schema.ts / mvu_schema.ts）

### 2.1 顶层模块划分

参照 2039 的七大模块思路，结合群穿题材调整为以下十大模块：

```
stat_data
├── 世界          # 时间轴 + 阶段标记 + 当前大事件
├── 基地车        # 基地车状态（能源/物资/设备/位置）
├── 科技树        # 技术扩散 / 知识红利（DAG 前置依赖 + 阶段推进 + 解锁联动）
├── 组织结构      # 政治局/军委/边区政府/各部门人事与运作
├── 军事          # 武装力量（编制/装备/弹药/训练）
├── 情报          # 谍报网络（渗透/反间谍/关键情报）
├── 外部势力      # 清廷/北洋/列强/革命党/本地势力/外交
├── 国内情势      # 控制/民生/治安/舆论/经济
├── 群友角色      # 94 名穿越群友群像追踪（z.record 动态键）
├── 非群友角色    # 本地人/清廷官员/其他 NPC 追踪（z.record 动态键，随剧情增长）
└── 事件队列      # 近期已发生事件 + 潜在危机
```

### 2.2 各模块字段设计要点

#### 模块一：世界

```ts
const 世界Schema = z.object({
  当前日期: z.string(),                    // 1905-05-10 起步，格式 YYYY-MM-DD
  当前时间: z.string(),                    // HH:MM
  政治阶段: z.string().describe('当前所处政治阶段，常见取值见 变量更新规则.yaml 的 政治阶段参考列表（穿越初混乱期/临时领导小组期/政治局常委会执政期/军管期/分裂动荡期/崩溃期），可按剧情自由扩展'),
  当前大事件: z.string(),                  // 一句话概括最紧迫事件
  穿越天数: z.coerce.number(),             // 自 1905.5.10 起的天数，便于 AI 推算时间
  // 注：原 当前会议 字段及 会议类型参考列表 已删除，推迟到后期扩展包（会议制度属中国政府运转逻辑深度建模）
});
```

#### 模块二：基地车（核心限制载体）

基地车是「不可复制的唯一资产」，其状态直接决定穿越者能力上限。

```ts
const 基地车Schema = z.object({
  当前位置: z.string(),                    // 如「安徽当涂县长江南岸」
  移动状态: z.string().describe('基地车移动状态，常见取值：驻扎/低速行军/故障停驶，可按剧情扩展'),
  反应堆功率等级: z.string().describe('反应堆功率档位，常见取值：满功率/降功率行军/低功率维持，可按剧情扩展'),
  配电系统负载: z.coerce.number().transform(v => _.clamp(v, 0, 100)),  // 0=空载 100=过载

  // 能源瓶颈是核心限制
  反应堆状态: z.string().describe('反应堆运行状态，常见取值：正常/降功率/故障/危急，可按剧情扩展'),
  配电系统状态: z.string().describe('配电系统状态，常见取值：正常/紧张/过载限流/故障，可按剧情扩展'),

  // 物资储备（键名自由扩展，支持玩卡时新增任意物资类型）
  // 初始内置：给养/燃料/医药/金属备件/现代枪械弹药/高产作物种子/电子设备，AI 可按剧情新增 建筑材料/纺织品/化工原料 等
  物资储备: z.record(
    z.string().describe('物资名，如 给养/燃料/医药/金属备件/现代枪械弹药/高产作物种子/电子设备/建筑材料/纺织品/化工原料，可按剧情自由扩展'),
    z.object({
      储备水平: z.string().describe('储备水平，常见取值见 物资储备水平参考列表（充足/充足但消耗中/紧张/危急/耗尽/完好/部分损失/污染风险/已损失），可按剧情扩展'),
      备注: z.string(),
    }),
  ),

  // 不可复制核心设备（键名自由扩展，损毁即永久失去）
  核心设备状态: z.record(
    z.string().describe('核心设备名，如 聚变堆/服务器/数控母机/图形工作站，可按剧情新增'),
    z.object({
      状态: z.string().describe('设备状态，常见取值：正常/降级/故障/损毁，可按剧情扩展'),
      备注: z.string(),
    }),
  ),

  // 武装与外骨骼（消耗性）
  外骨骼可用数量: z.coerce.number(),
  外骨骼备注: z.string(),                 // 记录磁体/电芯/芯片等不可补货瓶颈
});
```

#### 模块三：科技树（技术扩散 / 知识红利）

> **设计原因**：这是群穿叙事的**核心引擎**。94 名现代群友的最大资产不是基地车——基地车的物资会耗尽、设备会损坏——而是他们头脑中的**现代知识体系**。什么技术可以立即落地？什么需要前置科技？什么在 1905 年的物质条件下根本不可行？这些问题的答案直接决定穿越者政权的兴衰。
>
> **架构决策：结构硬编码 + MVU 只存运行时状态**
>
> 科技树的 DAG 结构（节点名/前置依赖/研发耗时/解锁能力/备注）是**世界观设定**，不应被 AI 修改。因此：
>
> | 存储位置 | 内容 | 可被 AI 修改？ |
> |----------|------|:---:|
> | `tech_tree.ts` | 静态结构：全部科技节点（14 个应用分支 × 三阶段 ~20 节点 + 系统性科研三阶段 25 节点）的所属分支/描述/前置科技/研发总耗时_天/解锁能力/备注 | ❌ |
> | MVU `stat_data.科技树` | 运行时状态：每个节点的已研发天数/启动时间/完成时间 + 研究方向选择 + 整体态势 | ✅ |
>
> 兜底脚本与状态栏通过 **join 两者**获取节点全貌：
>
> ```
> 节点全貌 = tech_tree.ts[节点名] + stat_data.科技树.科技节点[节点名]
> ```
>
> 新增/修改节点流程：先改 `科技树DAG指南.md`（人类讨论）→ 确认后同步改 `tech_tree.ts` → 无需改 Schema 或 initvar。
>
> **研发方向系统**：
>
> | 方向 | 数量 | 研发速率 | 说明 |
> |------|------|----------|------|
> | 主要研究方向 | 1 个（初始） | 100% | 全力推进。**每完成一个整合型节点（M1-M6 + 大规模科研计划共 7 个）+1 槽，终态 8 个** |
> | 次要研究方向 | 3 个（初始） | 30% | 碎片时间+兼职。**每完成一个系统性基础节点（S1-S18）+1 槽，终态 21 个** |
> | 被动研究方向 | 不限 | 5% | 闲聊/休息间无意推进 |

```ts
// ─── MVU 中只存储运行时状态 ──────────────────────────────
// 静态结构（所属分支/描述/前置科技/研发总耗时_天/解锁能力/备注）见 tech_tree.ts

const 科技节点运行时Schema = z.object({
  已研发天数: z.coerce.number().describe('等效研发天数（由兜底脚本规则 11b 按方向速率加权自动累加，AI 不手动修改）'),
  启动时间: z.string().describe('YYYY-MM-DD，启动研发的日期；空字符串=尚未启动'),
  完成时间: z.string().describe('YYYY-MM-DD，实际完成日期；由兜底脚本在 已研发天数 ≥ tech_tree.ts.研发总耗时_天 时自动填写'),
});

const 科技树Schema = z.object({
  // 运行时状态：键名必须与 tech_tree.ts 中定义的节点名一致
  科技节点: z.record(z.string().describe('科技节点名'), 科技节点运行时Schema),

  // 研究方向选择（玩家/GM 策略决策）
  主要研究方向: z.string().describe('当前全力推进的科技节点名（100%速率）；空字符串 "" = 未设定'),
  次要研究方向: z
    .array(z.string())
    .transform(data => _.take(data, 3))
    .describe('次要推进的科技节点名（最多3个，各30%速率）'),

  // 整体态势
  科技发展方针: z.string().describe('当前科技发展总方针，如 "优先保障生存"/"军事优先"/"民生与工业并重"'),
  技术保密等级: z.string().describe('整体技术保密策略：严格保密/选择性开放/逐步扩散'),
  科技自主度: z.string().describe('由兜底脚本规则 11d 自动计算，AI 不手动修改'),
});
```

**完整 DAG 图**：见 `科技树DAG指南.md`（14 个应用分支、每分支三阶段约 20 节点，外加系统性科研三阶段：18 个基础节点 + 6 个整合型节点 + 1 个终极节点）。**AI 能力可见性分级**（已解锁 / 可启动 / 不可见三层）的设计详见该指南 §0。

**研发方向选择示例**：


> **关键设计约束**：
>
> 1. **前置依赖硬约束**：兜底脚本规则 11a 读取 `tech_tree.ts` 的 `前置科技` 字段进行校验。若某节点 `已研发天数 > 0` 但其前置科技未完成，回滚增量并警告 AI。
> 2. **研发速率由方向决定**：主要=100%、次要=30%、被动=5%。兜底脚本规则 11b 按日历流逝天数自动累加。
> 3. **解锁联动**：兜底脚本规则 11c 在节点完成时读取 `tech_tree.ts` 的 `解锁能力` 字段，触发跨模块联动。
> 4. **科技自主度自动计算**：兜底脚本规则 11d 遍历 `tech_tree.ts`，统计已完成节点中不含「依赖基地车」标记的比例。
> 5. **状态栏三档时间预估**：读取 `tech_tree.ts` 的 `研发总耗时_天` + MVU 的 `已研发天数`，按方向速率计算预计完成日期。
> 6. **AI 不可新增节点**：`tech_tree.ts` 是硬编码常量，AI 无法在对话中 insert 新科技节点。新增节点需 GM 修改 `tech_tree.ts` 并重新部署脚本。

#### 模块四：组织结构（权力中枢，扁平化方案）

> **设计原则：高度自定义，适配反复改组**
>
> COT 说明给出的两阶段领导结构（5.10-5.13 临时领导小组 / 5.13 至今中央政治局常委会）只是**起点**。随着正文推进，穿越者政权必然经历多次体制演变：政治局常委会可能扩容/缩编/改名、军委与边区政府可能合并或拆分、可能新增「人民委员会」「战时内阁」「制宪会议」等全新机构、也可能撤销旧机构。因此**绝不能用固定字段硬编码机构名**，必须采用**动态机构表**：
>
> - 顶层用 `z.record(机构名, 机构Schema)`，AI 可自由 `insert`/`remove`/`rename` 任意机构
> - 采用**扁平化方案**：所有机构平铺在 `机构` record 中，通过 `上级机构` 字段表达层级（政治局 → 军委 → 教导连 → 一排 → 班），顶层机构 `上级机构=''`。真实层级最多约 5 层，无需 `z.lazy` 递归
> - `当前体制` 改为自由字符串，容纳任何新体制名
> - 用 `当前最高权力机构` 指针字段标记当前掌权的顶层机构键名，便于兜底脚本与状态栏快速定位
> - 政治局常委会不再是固定字段，而是 `机构` 中的一个键，其下可自由调整常委名单、增设办事组、分设书记处等

```ts
// ─── 扁平化机构 Schema（初期运行时核心集）────────────────────────────
// 设计原则（删减说明）：
```

**改组示例**：改组即操作 `机构` record 的键（`insert`/`rename`/`remove`）+ 更新 `当前最高权力机构` 指针 + 通过 `上级机构` 字段表达层级。如 5.13 建政插入政治局常委会/中央军委/边区政府/中央办公室，军委下再插入装备室/教导连，教导连下插入各排——通过 `上级机构` 字段串起层级。

> **注意事项**：
>
> 1. **兜底规则 1（职务↔角色）**：单层遍历 `机构` record，将每个机构的 `一把手`/`二把手`/`负责人`/`副职`/`成员` 同步到 `群友角色` / `非群友角色.某人.当前职务`，并自动追加履历。`所属系统` 已从角色记录中删除，改为状态栏渲染时实时从机构表推导（取 `上级机构` 链的顶层机构名）。
> 2. **兜底规则 2（死亡→空缺+履历锁定）**：角色死亡时，单层遍历 `机构` record 清空其职务，追加履历 + 锁定 `关键标签` 中的 `已死亡` 标签。人员死亡不可复活，兜底脚本应阻止 AI 将其改回非死亡状态。
> 3. **`当前体制` 不再用 enum**：因体制名无法穷举未来演变，改为 `z.string()`。若需限制 AI 乱填，可在 `变量更新规则.yaml` 中列举常见取值作为参考，但不做硬约束。
> 4. **中国政府运转逻辑深度建模（党政二元/双首长/条块/决策机制/人事/编制/财政/区划/监督/会议/保密）**：作为**后期扩展包**，建政 90 天后、控制区扩大后按需启用。初期机构 Schema 只保留高频字段 + `制度备注` 自由文本，低频制度字段合并其中，减轻 AI 每轮填写负担。

> **履历字段设计要点**：
>
> - **为什么需要履历**：94 人群像，AI 极易忘记某人已死亡/已升迁/已叛变/已失踪。履历以时间线冻结关键节点，AI 每轮填写前可快速回溯。
> - **自动追加**：兜底脚本应在规则 1（职务变更）、规则 2（死亡）、规则 8（派系斗争）触发时自动向对应角色的履历追加记录。
> - **关键标签**：由履历派生，便于状态栏快速筛选（如筛出所有"已死亡"角色、所有"叛变"角色）。兜底脚本在追加履历后同步更新关键标签，且锁定关键标签防止 AI 误删。
> - **保留最近 8 条**：普通群友长期只有 1 条，8 条足够回溯关键节点，过多占用上下文。

> **94 人群友初始化策略**：initvar.yaml 中需为所有 94 名群友生成初始记录。可编写一次性脚本从 `世界书/00-04` 各角色档案自动生成初始 stat_data，避免手写 94 份。初始 `履历` 填 `[{ 时间: '1905-05-10 00:00', 事件: '随基地车群穿至安徽当涂', 来源: 'GM' }]`，初始 `关键标签` 为 `[]`。**性别/年龄 写入 initvar**（防止 AI 乱写）；其余静态档案信息（穿越前身份/原群归属/所属派系/政治立场/效忠对象/更关心的事/个人剧情线）不写入 initvar，由世界书角色档案承载；initvar 中只填运行时字段（当前职务/位置/活动/人身安全等）。**仅有基本信息的角色**其世界书档案填入基本信息 + 「普通群友_通用模板」的默认倾向。
>
> **非群友角色初始化策略**：initvar.yaml 中初始为空 record `{}`，随剧情推进由 AI 自由 insert 新角色。首次出场时填写精简档案即可。

#### 模块九：军事（武装力量）

> **设计原因**：当前武装力量编制隐藏在 `机构` record 中（军委→教导连→排），但缺少独立的军事维度——武器装备清单、弹药储备（基地车弹药不可复制）、训练水平、士气。这些数据与「机构表中的职务任命」是正交关系：机构表管"谁带兵"，军事模块管"兵和枪的状态"。两者分开才能避免 Schema 臃肿，各司其职。
>
> **与机构表的关系**：机构表 `类型='军事指挥'` 的机构（如中央军委/教导连/某排）负责指挥链——谁是一把手、谁是二把手、当前任务；军事模块负责该部队的**战斗力状态**——兵力/装备/训练/士气。兜底脚本规则 10（军事）做双向联动。

```ts
// ─── 单支部队子 Schema ──────────────────────────────────

  备注: z.string(),
});

// ─── 武器装备记录 ────────────────────────────────────────

// ─── 军事顶层 ────────────────────────────────────────────

  // 弹药储备（基地车不可复制弹药，核心限制）
  现代弹药储备: z.string().describe('现代弹药储备水平，常见取值见 弹药储备水平参考列表（充足/消耗过半/紧张/危急/耗尽），可按剧情扩展'),
```

> **关键设计约束**：
>
> - **弹药不可复制**：基地车可造枪但不可造现代弹药（无化工产线），现代弹药储备是单向消耗指标。`充足 → 消耗过半 → 紧张 → 危急 → 耗尽` 的跌落不可逆，耗尽后群友只能用缴获抬枪/土造火铳/刺刀作战。
> - **部队键名 ↔ 机构表**：兜底脚本规则 10 确保 `军事.部队` 的键名与 `组织结构.机构` 中 `类型='军事指挥'` 的机构键名一致。AI insert 新部队时，兜底脚本自动在机构表创建同名军指机构；机构表 remove/rename 部队类机构时，兜底脚本同步更新军事模块。

#### 模块十：情报（谍报网络）

> **设计原因**：1905 年穿越者被清廷/列强/革命党环伺，情报是生存第一道防线。当前 `外部势力.对穿越者了解程度` 是**对方对我们的了解**（被动），情报模块追踪**我们对对方的了解 + 主动情报活动**（主动）。两者形成攻防闭环。

```ts
// ─── 情报网络节点 ────────────────────────────────────────

  最近情报: z.string(),

    z.string().describe('目标势力名，如 清廷中央/两江总督/安徽巡抚/北洋新军/日本/同盟会/本地士绅，与外部势力键名对应'),

  反间谍效果: z.string().describe('反间谍效果评估，常见取值：严密/有效/一般/有漏洞/形同虚设，可按剧情扩展'),

  情报准确度: z.string().describe('当前情报整体可靠性评估，常见取值：高/中/低/盲区，可按剧情扩展'),
```

> **关键设计约束**：
>
> - **情报网络键名 ↔ 外部势力**：`对外情报网络` 的键名应与 `外部势力` record 键名尽量一致，便于状态栏联动渲染（点击情报节点跳转到对应外部势力详情）。兜底脚本规则 11（情报联动）确保一致性。
> - **情报可靠性衰减**：若某情报节点长期未更新（穿越天数 > 其上次更新日 + 30），可靠性自动降级。AI 需要持续维护情报来源，否则会陷入"情报盲区"。
> - **反间谍 ↔ 外部势力了解程度**：反间谍效果越差 → 外部势力对穿越者的了解程度上升越快（兜底脚本规则 11）。

参照 2039 的 `deriveAll`，自动生成 `$` 前缀的高层提示标签：

| 派生字段 | 推导依据 | 用途 |

| `$整体危机等级`     | 治安 + 物资储备（给养等关键项）+ 外部威胁 + 潜在危机 | 综合危机评估                        |

| `$穿越者存活数`     | 各角色情况中非死亡人数                               | 94 人减员追踪（人员死亡不可复活！） |

| `$科技自主度`       | 成熟科技中不依赖基地车核心设备的比例                 | 提示 AI 脱离基地车后的能力上限      |
| `$已解锁能力清单`   | 遍历已完成节点，汇总 `tech_tree.ts.解锁能力`         | **AI 叙事可用能力的唯一来源**       |
| `$可启动的科技节点` | 前置科技已全部完成的未完成节点 + 其解锁能力预览      | 供 AI 选择研究方向，远期节点不可见  |

> **科技能力可见性分级（核心设计）**：科技树的静态结构（含每个节点的 `解锁能力` 字段）硬编码在 `tech_tree.ts`，**不进 MVU、不进 AI 上下文**。若把所有节点的解锁能力塞进 MVU，AI 会看到未解锁能力并误用；若完全不暴露，AI 又不知道当前能做什么。因此用两个派生字段做**三层信息分级**：
>
> | 层级 | 派生字段 | 内容 | AI 可见性 |
> |------|----------|------|-----------|
> | 层1 已解锁 | `$已解锁能力清单` | 已完成节点的解锁能力汇总 | ✅ AI 叙事可用能力的唯一来源 |
> | 层2 可启动 | `$可启动的科技节点` | 前置已满足的未完成节点 + 解锁能力预览 | ✅ AI 据此选择研究方向 |
> | 层3 不可见 | （无） | 前置未满足的远期节点 | ❌ AI 完全看不到，零干扰 |
>
> 这样 AI 每轮看到的是「当前已解锁的能力清单」+「接下来可研发的候选节点及其能力预览」，既不会遗漏已解锁能力，也不会被远期未解锁能力干扰判断。状态栏 `TechTreePanel` 仍展示完整 DAG 供玩家/GM 决策，但那是给 USER 看的，不进 AI 上下文。

> `$财政健康度 / $监督有效性 / $地方控制度 / $制衡机制状态` 依赖低频字段（三权指针/监督机构/区划关系/财政审计），低频字段不填时派生不出有意义结果，推迟到后期扩展包。

```ts
export function deriveAll<T extends RuntimeMvuData>(data: T) {
  return {
    ...data,

    $基地车可用度: derive基地车可用度(data),
    $外部威胁等级: derive外部威胁等级(data),
    $整体危机等级: derive整体危机等级(data),
    $当前剧情焦点: derive当前剧情焦点(data),
    $穿越者存活数: derive穿越者存活数(data),    // 仅统计 群友角色 中非死亡人数（非群友不计入）
    $军事实力指数: derive军事实力指数(data),
    $情报覆盖度: derive情报覆盖度(data),
    $科技自主度: derive科技自主度(data),        // 成熟科技中 所需基础设施 不含「基地车」的比例
    $已解锁能力清单: derive已解锁能力清单(data), // 已完成节点的解锁能力汇总——AI 叙事可用能力的唯一来源
    $可启动的科技节点: derive可启动的科技节点(data), // 前置已满足的未完成节点+解锁能力预览，供 AI 选方向
  };
}

// ─── 科技能力可见性分级（核心设计）──────────────────────────────
// 问题：科技树静态结构（含 解锁能力 字段）在 tech_tree.ts 硬编码，不进 MVU、不进 AI 上下文。
//      若把所有节点的解锁能力塞进 MVU，AI 会看到未解锁能力并误用；若完全不暴露，AI 又不知道当前能做什么。
// 解决：用两个派生字段做三层信息分级——
//   层1 已解锁：$已解锁能力清单（已完成节点的能力）→ AI 叙事可用能力的唯一来源
//   层2 可启动：$可启动的科技节点（前置已满足的未完成节点+能力预览）→ AI 据此选研究方向
//   层3 不可见：前置未满足的节点 → AI 完全看不到，零干扰
// 注：派生函数需 import { 科技树 } from '../../tech_tree'

/** 层1：遍历已完成节点，汇总解锁能力，每条带来源标注便于 AI 溯源 */
function derive已解锁能力清单(data: RuntimeMvuData): string[] {
  const 清单: string[] = [];
  for (const [节点名, 状态] of Object.entries(data.科技树.科技节点)) {
    const 定义 = 科技树[节点名];
    if (!定义) continue;
    // 已完成 = 已研发天数达标 且 完成时间非空（完成时间由兜底脚本规则 11b 填写）
    if (状态.已研发天数 >= 定义.研发总耗时_天 && 状态.完成时间) {
      for (const 能力 of 定义.解锁能力) {
        const 条目 = `[${定义.所属分支}] ${能力}（${节点名}，${状态.完成时间}完成）`;
        if (!清单.includes(条目)) 清单.push(条目);
      }
    }
  }
  return 清单;
}

/** 层2：前置科技已全部完成、但本节点尚未完成的节点，附解锁能力预览 */
function derive可启动的科技节点(data: RuntimeMvuData): Array<{
  节点名: string;
  分支: string;
  研发总耗时_天: number;
  已研发天数: number;
  解锁能力: string[];
}> {
  const 节点状态 = data.科技树.科技节点;
  const 已完成 = (名: string) => {
    const s = 节点状态[名];
    const d = 科技树[名];
    return !!s && !!d && s.已研发天数 >= d.研发总耗时_天 && !!s.完成时间;
  };
  const 结果 = [];
  for (const [节点名, 定义] of Object.entries(科技树)) {
    const 状态 = 节点状态[节点名];
    if (!状态) continue;                       // MVU 中无记录=未启动，但需前置满足才可启动
    if (状态.完成时间) continue;               // 已完成 → 进层1
    // 前置科技必须全部已完成
    const 前置全满足 = 定义.前置科技.every(已完成);
    if (!前置全满足) continue;                 // 前置未满足 → 层3，AI 不可见
    结果.push({
      节点名,
      分支: 定义.所属分支,
      研发总耗时_天: 定义.研发总耗时_天,
      已研发天数: 状态.已研发天数,
      解锁能力: 定义.解锁能力,
    });
  }
  return 结果;
}
```

#### 规则 11：科技树联动（前置依赖硬约束 + 研发时间累加 + 解锁联动 + 科技自主度自动计算）

```ts
// ─── 11a：前置依赖硬约束（核心约束，不可绕过）───────────
// 遍历 科技树.科技节点：
//   若某节点 variables.已研发天数 > 0 且 variables_before.已研发天数 = 0（本轮新启动）

//     若任一前置科技.已研发天数 < 前置科技.研发总耗时_天（尚未完成）
//       → 回滚已研发天数增量，console.warn "前置科技 ${name} 尚未完成，${本节点} 不可启动"
//   若通过校验 → 允许启动
// 注：不可跳过前置科技——这是科技树的核心约束

// ─── 11b：研发时间自动累加（本规则的核心）─────────────
// 计算本轮日历流逝天数 = 世界.当前日期 - 上一轮日期（从 variables_before 推算）
// 若流逝天数 ≤ 0 → 跳过（同一轮重复触发或无时间推进）
//
// 遍历 科技树.科技节点：
//   若 已研发天数 ≥ 研发总耗时_天 → 跳过（已完成）
//   若 启动时间 = '' → 跳过（未启动）
//   若 节点名 = 主要研究方向 → 增量 = 流逝天数 × 1.0
//   若 节点名 ∈ 次要研究方向 → 增量 = 流逝天数 × 0.3
//   否则（被动） → 增量 = 流逝天数 × 0.05
//   已研发天数 += 增量
//   若 已研发天数 ≥ 研发总耗时_天（本轮新完成）：
//     → 完成时间 = 世界.当前日期
//     → 触发规则 11c（解锁联动）
//     → 事件队列追加「科技突破：${节点名} 研发完成」
//
// 注：不设时间上限——研发严格跟随日历流逝。若叙事跳过了 30 天，
//     研发就推进 30 等效天（按速率加权），与故事时间保持严格一致。
//     防止玩家纯靠"等待"跳过所有研发是 GM 的职责（外部威胁/物资消耗/
//     历史大事件不会等人），而非兜底脚本的职责。

// ─── 11c：科技完成 → 解锁联动 ────────────────────────────
// 当 已研发天数 ≥ 研发总耗时_天 首次达成时：
//   → 遍历该节点的 解锁能力 数组，对每条能力标签做联动：
//     - 能力含 '可生产'/'可制造' → 在 军事.武器装备 中提示可新增对应装备条目
//     - 能力含 '通信'/'电报' → 在 情报 模块追加提示事件
//     - 能力含 '电力'/'发电' → 在 基地车.配电系统负载 追加备注 "可由独立发电分担"
//     - 能力含 '医药'/'外科' → 在 群友角色 相关角色.备注 追加
//   → 事件队列追加「科技突破：${节点名} 研发完成」

// ─── 11d：科技自主度自动计算 ──────────────────────────────
// 统计 科技树.科技节点 中 已研发天数 ≥ 研发总耗时_天（已完成）的节点总数 = 总完成数
// 统计上述节点中 备注 不含「依赖基地车」标记的数量 = 非依赖数
//   （由 GM/玩家在 备注 中添加「依赖基地车」标记来标注哪些技术强依赖基地车设备）
// 科技树.科技自主度 按比例映射：
//   总完成数 = 0 → '完全依赖'
//   非依赖数/总完成数 < 0.2 → '完全依赖'
//   0.2-0.4 → '重度依赖'
//   0.4-0.6 → '部分自主'
//   0.6-0.8 → '基本自主'
//   > 0.8 → '完全自主'
// 注：此计算每轮自动执行，AI 不手动修改

// ─── 11e：AI 误用未解锁能力检测（防干扰核心）─────────────
// 问题：即使有 $已解锁能力清单，AI 仍可能在叙事中凭空使用未解锁能力
//      （如蒸汽机未完成却写"用蒸汽机驱动锻锤"）。
// 检测时机：VARIABLE_UPDATE_ENDED 后，读取本轮 AI 输出文本（若可获取）
// 检测逻辑：
//   1. 构建本轮"已解锁能力关键词表"——遍历 $已解锁能力清单 提取关键名词
//      （如 蒸汽机/电报/抗生素/小高炉 等）
//   2. 构建本轮"未解锁但易误用关键词表"——遍历 tech_tree.ts 中所有未完成节点
//      的 解锁能力 + 节点名，提取关键名词
//   3. 扫描 AI 输出文本：
//      - 若出现"未解锁关键词"且上下文非"讨论/规划/否定"（如"我们还没蒸汽机"）
//        → console.warn "AI 可能误用未解锁能力：${关键词}（对应节点 ${节点名} 未完成）"
//      - 同时在事件队列追加一条 GM 提示事件：
//        { 事件: '⚠️能力误用警告：AI 叙事中使用了未解锁的 ${关键词}', 来源: '兜底脚本' }
//   4. 此规则仅警告不回滚——AI 输出文本无法回滚，但警告会进入下一轮上下文

// 注：关键词提取需维护一个"能力名词→节点名"映射表（可从 tech_tree.ts 自动构建），
//      避免误报（如"蒸汽"出现在"蒸汽浴室"中不应触发蒸汽机警告）。
```

```ts
// 扁平化方案下所有机构平铺在 机构 record 中，通过 上级机构 字段表达层级。
// 单层遍历即可覆盖所有机构；如需树状展示，按 上级机构 字段在内存中重建树。
function walk机构(
  机构表: Record<string, 机构Node>,
  回调: (机构名: string, 机构: 机构Node) => void,
) {
  for (const [机构名, 机构] of Object.entries(机构表)) {
    回调(机构名, 机构);
  }
}

// 收集某角色在机构表中担任的所有职务（用于规则 1 反向同步与规则 2 清空）
function collect角色职务(
  机构表: Record<string, 机构Node>,
  角色名: string,
): Array<{ 机构名: string; 角色: '一把手' | '二把手' | '副职' | '成员' }> {
  const 结果 = [];
  walk机构(机构表, (机构名, 机构) => {
    if (机构.一把手 === 角色名) 结果.push({ 机构名, 角色: '一把手' });
    if (机构.二把手 === 角色名) 结果.push({ 机构名, 角色: '二把手' });
    if (机构.副职?.includes(角色名)) 结果.push({ 机构名, 角色: '副职' });
    if (机构.成员?.includes(角色名)) 结果.push({ 机构名, 角色: '成员' });
  });
  return 结果;
}

// 取某机构的顶层机构名（沿 上级机构 链向上回溯，用于状态栏推导"所属系统"）
function 顶层机构(机构表: Record<string, 机构Node>, 机构名: string): string {
  let 当前 = 机构名;
  while (机构表[当前]?.上级机构) {
    当前 = 机构表[当前].上级机构;
  }
  return 当前;
}
```

```yaml
        - 进行中节点（已研发天数>0）的进度也可在此查看，用于判断是否调整方向
    _AI能力使用总约束:
      说明: 科技能力可见性分级的核心约束，防止 AI 凭空使用未解锁能力
      规则:
        - 叙事中角色"能做什么"的唯一依据是 $已解锁能力清单，不得超出
        - 讨论未来规划时可提及 $可启动的科技节点 中的节点及其能力预览
        - 前置未满足的远期节点（不在上述两个派生字段中）不得在叙事中作为已实现能力使用
          （如"用蒸汽机驱动锻锤"而蒸汽机未完成属违规；"我们计划研发蒸汽机"属合规）
        - 兜底脚本规则 11e 会扫描 AI 输出检测误用并警告，GM 可据此干预
      check:
        - 必须是 科技节点 record 中存在的键名，或为空字符串 ''（未设定）
        - 该节点必须 `已研发天数 < 研发总耗时_天`（未完成）且前置科技已完成
        - 获得 100% 研发速率
    次要研究方向:
      check:
        - 最多 3 个，必须是 科技节点 record 中存在的键名
        - 每个节点必须 `已研发天数 < 研发总耗时_天`（未完成）且前置科技已完成
        - 各获得 30% 研发速率
        - 不可与 主要研究方向 重复
    科技节点.${节点名}:
      type: |-
        {
          已研发天数: number;     // 等效研发天数（由兜底脚本规则 11b 自动累加，AI 不应手动修改）
          启动时间: string;       // YYYY-MM-DD，启动研发的日期；空=未启动
          完成时间: string;       // YYYY-MM-DD（由兜底脚本在完成后自动填写）
        }
      check:
        - 科技节点名即 record 键名，键名必须在 tech_tree.ts 中已定义（硬编码约束，AI 不可 insert 新节点）
        - 前置依赖/研发耗时见 tech_tree.ts，AI 不可修改
        - 已研发天数 由兜底脚本规则 11b 按方向速率自动累加，AI 不应手动修改
        - 启动时间 在 AI 首次将该节点设为主要/次要研究方向时由 AI 填写
        - 完成时间 由兜底脚本在 已研发天数 ≥ tech_tree.ts 中该节点的 研发总耗时_天 时自动填写
        - AI 若需要新增科技节点，须通知 GM 修改 tech_tree.ts 并重新部署

  事件队列:
    近期已发生事件:
      check:
        - 每轮至少追加 1 条本轮关键事件
        - 保留最近 8 条
        - 影响 字段须含 1-2 个受影响方面
    潜在危机:
      check:
        - 保留最近 6 条
        - 触发条件须具体可判，如「清廷调兵超过 500 人」
```

### 5.4 关键组件

- `WorldSection.vue`：世界状态栏，展示日期/政治阶段/当前大事件/穿越天数
- `BaseVehicleBar.vue`：基地车状态条
- `TechTreePanel.vue`：科技树面板（核心）。顶部状态条展示 科技自主度（大徽标，完全依赖=红/基本自主=绿）+ 科技发展方针 + 技术保密等级。其下为**研究方向选择区**：下拉框选择主要研究方向 + 多选次要研究方向（最多3个），每个选项旁显示该方向的预计完成日期。主体分为：
  - **DAG 可视化区**：按 `前置科技` 字段在内存中重建 DAG 图，用 CSS flex/grid 渲染分层依赖关系。前置科技未满足的节点灰显+锁图标；已完成的节点绿色边框+解锁能力标签；进行中的节点进度条（百分比 = 已研发天数 / 研发总耗时_天 × 100），进度条颜色按方向区分（主要=金色/次要=蓝色/被动=灰色）
  - **三档时间预估**：每个进行中节点旁显示三档预计完成日期——「主：YYYY-MM-DD」「次：YYYY-MM-DD」「被动：YYYY-MM-DD」——供玩家做方向决策
  - **分支进度概览**：按 所属分支 分组，每分支展示 已完成节点数/总节点数 进度条
  - **节点详情卡片**：点击节点展开——前置科技链（递归展示上游依赖）、研发进度条+剩余天数、产出描述、解锁能力标签、启动时间/完成时间、备注
- `OrganizationTree.vue`：组织结构树。按 `上级机构` 字段在内存中建树后渲染，顶部高亮 `当前最高权力机构`，已撤销机构灰显。**注意：`运作状态` 为自由字符串，着色时需对未知取值做 fallback（默认色）**。每个机构节点展示一把手/二把手、运作状态、权势指数、实有人数；`制度备注` 非空时折叠展示
- `CharacterGrid.vue`：角色群像网格（核心，需支持搜索/筛选/排序）。分两个 Tab 分别展示「群友」（94 人固定）与「非群友」（动态增长）。按所属系统筛选时需实时从机构表推导（沿 `上级机构` 链取顶层机构名）。每个角色卡片展示关键标签徽标（已死亡/叛变/重伤等），履历通过悬浮 tooltip 或展开行展示最近 5 条
- `ExternalForcesPanel.vue`：外部势力面板。以表格/卡片对比各势力的态度+了解程度+我方外交策略，`内部分支` 非空时折叠展示可拉拢人物。微观本地势力（士绅/宗族/会党等）与宏观势力同表展示，可按 `在当地影响力` 或键名前缀分组
- `MilitaryPanel.vue`：军事面板。顶部状态条展示总兵力+现代弹药储备（色阶警示），下方按部队分卡片展示兵力/训练水平/士气（表情图标）/当前位置/当前任务。武器装备清单以表格展示数量/来源/状态。弹药储备='耗尽' 时顶部红色警示
- `IntelligencePanel.vue`：情报面板。反间谍措施与效果徽标，情报准确度大徽标（绿/黄/橙/红对应高/中/低/盲区）。对外情报网络以节点卡片展示（目标势力→来源→可靠性→最近情报），可靠性='不可靠' 或 '反间嫌疑' 时卡片边框警示色
- `EventQueue.vue`：事件队列

> `FinancePanel / AdminDivisionPanel / PersonnelPanel` 推迟到后期扩展包。

## 七、关键注意事项

### 7.1 角色管理的特殊处理

- **群友与非群友严格分离**：`群友角色` 记录 94 名穿越者（固定身份，有原群归属），`非群友角色` 记录本地人/NPC（随剧情动态增长）。两者 Schema 不同，AI 不会混淆。
- **不要用数组**，全部用 `z.record(z.string().describe('姓名'), ...)`，姓名为键
- **未出场角色保留原值**，AI 不应每轮重写所有角色，仅在 `变量更新规则.yaml` 中要求「本轮出场角色完整更新，被提及角色仅更新关系动态」
- **死亡不可复活**：`人身安全 = '已死亡'` 后，兜底脚本应阻止 AI 将其改回，并锁定 `关键标签` 中的 `已死亡` 标签
- **履历是防误判的核心**：AI 叙事跨度长、角色多，极易遗忘某人已死亡/已升迁/已叛变。履历以时间线冻结关键节点；兜底脚本在职务变更、死亡、派系变动时自动追加履历；AI 每轮填写前必须先读履历。`关键标签` 由履历自动派生并锁定，便于状态栏快速筛选
- **普通群友默认倾向**：无详细档案的角色，initvar 中填入「普通群友_通用模板」的默认行为倾向，AI 在倾向范围内选取不同侧面

### 7.2 基地车限制的硬约束

- 核心设备（聚变堆/服务器/数控母机/图形工作站）**不可拆卸、不可仿制、损毁即永久失去**
- 配电系统是真正瓶颈，生产必须切断生活区照明
- 外骨骼瓶颈在磁体/电芯/芯片，机械损坏可修，三者烧毁即报废
- 高产作物种子（物资储备中）一旦丢失或被本地作物杂交污染即不可恢复；其他物资消耗品可通过本地采购/缴获等方式补充
- 这些硬约束应在 `变量更新规则.yaml` 中明确告知 AI，并在兜底脚本中阻止 AI 违规「修复」已损毁核心设备

### 7.3 历史大事件按节奏发生

- 1905.9.5 日俄战争结束、9.2 废除科举、8.20 同盟会成立等，**按历史节奏发生，穿越者只能顺应或干预**
- 兜底脚本规则 7 应在日期推进到关键节点时自动触发事件
- 穿越者干预可能改变后续走向，但初始大事件不可跳过

### 7.4 派系与私交的区分

- 穿越前群聊关系（老纳垢群/新冰镐群/长烟群/割席独立）是**私交底色**，不代表政治立场一致
- 男权思想圈、跑团互动圈、狐朋狗友等是**关系圈**，可叠加在群归属之上
- 割席独立（脉冲/SCV/苟赤色/小杨生煎）因政见不同完全独立，不属于任何群
- 三叶（女）被全体新冰镐群成员厌恶，是特殊负面关系
- 这些关系应在世界书角色档案中体现（原群归属/所属派系/关系圈等静态属性），运行时演变（叛变/倒戈/派系重组）由 `履历` 记录节点、`关键标签` 派生当前状态，并在 `变量更新规则.yaml` 中提示 AI 据此演绎互动

### 7.5 USER 摄像机视角

- USER 不扮演任何角色，不介入剧情
- 状态栏应呈现**客观全景**，不附加主观评判
- AI 回复应为群像叙事，多角色平行故事线，不聚焦单一主角
- `变量更新规则.yaml` 中应提示 AI「以摄像机视角记录多个角色的平行故事线」

### 7.6 中国政府运转逻辑（后期扩展包）

党政二元结构/双首长/条块关系/决策-执行-监督三权制衡/财政硬约束/人事任命权/议事协调机构与领导小组/会议制度/请示报告制度/保密等级/行政区划博弈/字段更新频率分层——这一整套深度建模作为**后期扩展包**，建政 90 天后、控制区扩大后按需启用。初期机构 Schema 只保留高频字段 + `制度备注` 自由文本，减轻 AI 每轮填写负担。

## 八、低优先级扩展备忘（独立悬浮窗脚本）

以下维度在群穿叙事中具有潜在价值，但**不纳入 MVU 体系**——它们低频、偏 GM/玩家侧、且会让 AI 每轮填写负担过重。改为做成**独立的悬浮窗脚本**，单独存储、单独展示，不参与 `VARIABLE_UPDATE_ENDED` 兜底校验流水线。

### 8.1 设计定位

| 维度                                             | 是否进 MVU | 原因                                                |
| ------------------------------------------------ | ---------- | --------------------------------------------------- |
| 世界/基地车/科技树/组织/军事/情报/外部/角色/事件 | ✅ 进 MVU   | 高频、AI 每轮需读写的核心状态                       |
| 工业生产                                         | ❌ 不进     | 低频、偏 GM 记账、塞进 stat_data 会拖累 AI 每轮填写 |
| 自然环境                                         | ❌ 不进     | 季节/灾害由日期推算即可，无需每轮持久化             |
| 法律与制度                                       | ❌ 不进     | 法令文本长、低频、适合 GM 手动维护                  |
| 医疗与公共卫生                                   | ❌ 不进     | 伤员已在 `群友角色.人身安全` 追踪；疫情属偶发事件   |
| 教育与宣传                                       | ❌ 不进     | 长期累积型数据，AI 每轮维护成本高、收益低           |

**核心原则**：MVU 只承载「AI 每轮必须读写」的高频核心状态；低频、累积型、GM 侧的数据走独立悬浮窗，互不污染。

### 8.2 悬浮窗脚本架构

```
群友群穿/                         # 数据文件（根目录）
src/群友群穿/                     # 代码文件（在 src/ 下）
└── 脚本/
    └── 扩展备忘悬浮窗/
        ├── index.ts          # 入口：注册悬浮窗、挂载 Vue 应用
        ├── store.ts          # 独立存储（脚本变量 / 聊天变量，非 MVU）
        ├── App.vue           # 悬浮窗主面板（多 Tab）
        └── components/
            ├── IndustryPanel.vue      # 工业生产
            ├── EnvironmentPanel.vue   # 自然环境
            ├── LawPanel.vue           # 法律与制度
            ├── MedicalPanel.vue       # 医疗与公共卫生
            └── EducationPanel.vue     # 教育与宣传
```

**存储方案**（二选一，推荐脚本变量）：

- **脚本变量**（`extension_settings` / 自定义键）：随角色卡脚本生命周期存取，不进 MVU `stat_data`，不触发 `VARIABLE_UPDATE_ENDED`
- **聊天变量**：若需跨楼层持久化且与聊天进度绑定，可用聊天变量；但会随聊天存档增大

> **关键**：悬浮窗脚本**不注册 MVU Schema**、**不监听 `Mvu.events.VARIABLE_UPDATE_ENDED`**、**不写入 `stat_data`**。它与 MVU 体系完全解耦，仅通过悬浮窗 UI 供 GM/玩家查看与手动编辑。

### 8.4 与 MVU 体系的边界规则

1. **只读不写**：悬浮窗脚本可只读读取 MVU `stat_data`（如日期、机构表、部队表、角色人身安全）用于联动展示，但**绝不回写** `stat_data`，避免双数据源冲突
2. **不参与兜底**：悬浮窗数据**不触发**任何 MVU 兜底规则（规则 1-16 均不扫描悬浮窗数据）
3. **不进 AI 提示词**：`变量更新规则.yaml` **不包含**悬浮窗字段的 check 条目；AI 每轮**不填写**悬浮窗数据
4. **GM 维护**：悬浮窗数据由 GM/玩家手动维护，或由 GM 在叙事后手动追加；AI 可在 narrative 中引用这些设定，但不负责持久化更新
5. **可选注入**：若希望 AI 感知悬浮窗数据，可由悬浮窗脚本在 `GENERATION_STARTED` 前将关键字段以 `inject` 方式注入提示词（参照 `@types/iframe/event.d.ts` 的注入接口），但这是可选行为，非默认

### 8.5 何时升级为 MVU 模块

当某个维度的数据满足以下**任一**条件时，可考虑从悬浮窗升级为 MVU 独立模块（届时参照后期扩展包的 `.optional()` + 按数据存在性触发策略）：

- AI 在叙事中**频繁需要读取**该维度才能合理推进剧情（如工业产能成为外交谈判筹码）
- 该维度数据**每轮都可能变化**且 AI 遗漏会导致剧情矛盾（如疫情扩散速度）
- 该维度与其他 MVU 模块存在**强联动**需要兜底脚本自动校验（如法令颁布→机构运作状态变化）

升级时将该维度的 Schema 追加到 `mvu_schema.ts`（标 `.optional()`），在 `变量更新规则.yaml` 补 check 条目，在兜底脚本补联动规则，悬浮窗对应 Tab 改为只读展示（数据源切换为 MVU）。
